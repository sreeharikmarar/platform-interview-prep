# Talk Tracks

## Q: Explain this in one minute.

**Answer:**

The Kubernetes scheduler is a specialized controller that watches for unscheduled pods (nodeName empty) and assigns them to nodes through a three-phase pipeline: filter, score, and bind. The filter phase eliminates infeasible nodes by running predicate plugins like NodeResourcesFit which checks if the node has enough CPU and memory for the pod's requests, TaintToleration which excludes nodes with taints the pod doesn't tolerate, and NodeAffinity which validates required node label selectors. Remaining feasible nodes enter the score phase where plugins like BalancedAllocation and ImageLocality rank nodes from 0-100, scores are summed with weights, and the highest-scoring node is selected. The bind phase runs asynchronously, creating a Binding object that updates the pod's spec.nodeName, then kubelet on that node sees the pod and starts containers. The scheduler is pessimistic - it makes decisions based on cluster state at scheduling time which can change before containers start. Scheduling is decoupled from pod lifecycle management - controllers create and delete pods, the scheduler only assigns nodes, enabling pluggable custom schedulers without modifying other components.

## Q: Walk me through the internals.

**Answer:**

The scheduler's Informer watches for pods with spec.nodeName empty and status.phase Pending. These pods are enqueued into a priority queue organized as activeQ for ready-to-schedule, backoffQ for retry with exponential backoff, and unschedulableQ for pods waiting on cluster state changes. When a pod is dequeued, the scheduler takes a consistent snapshot of cluster state - all nodes with their capacity and allocatable resources, and all running pods - to avoid race conditions during the scheduling cycle. Filter plugins run sequentially for each node: NodeResourcesFit sums resource requests from all running pods plus the new pod's requests and compares against node allocatable, TaintToleration matches pod tolerations against node taints, NodeAffinity evaluates requiredDuringScheduling label selectors, PodTopologySpread checks if placing the pod violates maxSkew constraints, and VolumeBinding validates that required PVs can bind to the node. Nodes that pass all filters enter the score phase where each scoring plugin assigns 0-100 points - BalancedAllocation prefers nodes with balanced CPU and memory usage to avoid resource waste, ImageLocality gives bonus points if the node already has the container image cached, and InterPodAffinity scores based on preferred pod affinity rules. Plugin scores are multiplied by configured weights and summed. The highest-scoring node is selected, with random tiebreaking. The Reserve plugin updates resource accounting, then binding moves to an async goroutine where PreBind plugins mount volumes, the Bind plugin POSTs a Binding object to the API server which updates pod.spec.nodeName, and PostBind plugins log metrics.

## Q: What can go wrong?

**Answer:**

The highest-impact failure is resource fragmentation where the cluster has aggregate capacity but no single node has enough for a pod. Symptoms are pods stuck Pending with events saying "0/5 nodes available: 3 Insufficient cpu, 2 Insufficient memory" even though total available CPU across nodes exceeds the pod's request. This happens when many small pods fragment resources or a pod requests more than any node's allocatable. The blast radius cascades if critical system pods like CoreDNS can't schedule, making the cluster unhealthy. Mitigation involves right-sizing resource requests, using PriorityClass so critical pods preempt lower-priority workloads, and running cluster autoscaler to add nodes, though autoscaler won't help if pods request more than single-node capacity. Second failure is taint-toleration mismatch causing cluster deadlock. If all nodes get tainted for dedicated workloads but pods lack corresponding tolerations, nothing can schedule. This commonly happens during cluster upgrades when new nodes have taints but workload manifests haven't been updated. Detection is through pod events showing "node(s) had untolerated taint" and mitigation is keeping untainted nodes for general workloads or using PreferNoSchedule instead of NoSchedule for soft isolation. Third failure is unsatisfiable pod affinity where a pod requires co-location with another pod that doesn't exist, or anti-affinity saturates when all nodes already have an instance and topologyKey is kubernetes.io/hostname, blocking the next pod.

## Q: How would you debug it?

**Answer:**

I start with kubectl describe pod to read events, which show the scheduler's filter results like "0/5 nodes available: 3 Insufficient cpu, 2 node(s) had untolerated taint." For resource issues, I run kubectl top nodes to see actual utilization versus requests, then kubectl describe nodes and grep for "Allocated resources" to see per-node breakdowns of requests versus allocatable. I check the pod's resource requests with kubectl get pod -o jsonpath for spec.containers[].resources.requests to see if they're unreasonably large. For taint issues, I list all node taints with kubectl get nodes -o json piping to jq to extract spec.taints, compare them against the pod's spec.tolerations, and identify which taint blocked scheduling. For affinity failures, I examine the pod's spec.affinity rules to see if podAffinity references label selectors that don't match any running pods, or if nodeAffinity requires node labels that don't exist. I use kubectl get pods -o wide with label selectors to visualize pod distribution for anti-affinity saturation. For PersistentVolume conflicts, I check the PV's nodeAffinity with kubectl get pv -o yaml and verify the pod's constraints are compatible with the volume's zone restrictions. I also review scheduler logs if available to see detailed filter and score results.

## Q: How would you apply this in a platform engineering context?

**Answer:**

In multi-tenant platforms, scheduler configuration is critical for isolation and efficiency. I use taints and tolerations to dedicate node pools - GPU nodes tainted with hardware=gpu:NoSchedule ensure only ML workloads with matching tolerations consume expensive resources, while general workloads stay on cheaper nodes. For tenant isolation, I taint nodes with tenant=acme:NoExecute so that only Acme's pods with the toleration can schedule there, preventing noisy neighbor issues. PriorityClasses establish a hierarchy where platform infrastructure pods like ingress controllers and monitoring agents have high priority and preempt lower-priority tenant workloads when resources are scarce, ensuring the platform remains operational. Pod anti-affinity on ingress controllers with topologyKey kubernetes.io/hostname spreads replicas across nodes for high availability, and zone anti-affinity spreads across availability zones for resilience to zone failures. TopologySpreadConstraints enforce even tenant distribution - if Tenant A's pods should spread across nodes with maxSkew 1, the scheduler prevents hot spots. For cost optimization, I use node affinity to prefer preemptible or spot instances for batch workloads while requiring on-demand instances for latency-sensitive services. I monitor pending pod metrics and scheduler queue depth to detect capacity issues early, triggering autoscaler before tenants hit failures.

## Q: Explain the difference between node affinity and taints/tolerations.

**Answer:**

Node affinity is pod-centric - the pod declares which nodes it wants via label selectors, with requiredDuringScheduling as hard constraints that block scheduling if unsatisfied, and preferredDuringScheduling as soft preferences that influence scoring. It's a pull model where pods select nodes. Taints and tolerations are node-centric - the node declares it repels pods via taints, and only pods with matching tolerations can schedule there. It's a push model where nodes reject pods. The use cases differ: node affinity is for workload requirements like "this pod needs SSD storage so match disktype=ssd label," while taints are for node characteristics like "this node is dedicated to GPU workloads, repel everything else." Taints have effects that node affinity doesn't - NoExecute evicts existing pods without tolerations, not just blocks new scheduling. You often combine them: taint GPU nodes with hardware=gpu:NoSchedule to repel general workloads, then GPU pods use both a toleration to bypass the repulsion and node affinity to ensure they schedule on GPU nodes not just anywhere that tolerates them.

## Q: How does preemption work and when should you use it?

**Answer:**

Preemption activates when no nodes pass filters for a high-priority pod. The scheduler simulates removing lower-priority pods from each node - a pod is a victim candidate if its priority value from PriorityClass is less than the pending pod's priority. After virtually removing victims, the scheduler re-runs filters to see if the node now passes. If yes, it scores candidate nodes preferring those requiring fewer evictions and lower total victim priority. The scheduler sets the pending pod's status.nominatedNodeName to reserve the chosen node, then deletes victim pods respecting their terminationGracePeriodSeconds. Crucially, the scheduler doesn't immediately bind - it waits for victims to terminate and the node to pass filters naturally, then schedules through the normal path. Use preemption for critical infrastructure - platform services like ingress controllers, monitoring agents, and DNS should have high priority so they preempt tenant batch jobs during resource pressure, keeping the platform operational. Never give high priority to all tenant workloads or they'll preempt each other chaotically. The priority hierarchy should be platform infrastructure highest, then latency-sensitive tenant services, then tenant batch jobs lowest. Monitor preemption events to detect if it's happening frequently, which indicates insufficient capacity or mis-sized resource requests.

## Q: How do topology spread constraints differ from pod anti-affinity?

**Answer:**

Pod anti-affinity is binary - either the pod can schedule or it can't based on whether other matching pods exist in the topology domain. It doesn't consider distribution imbalance. If you have 10 web pods and use anti-affinity with topologyKey kubernetes.io/hostname, the 11th pod fails if all nodes have at least one web pod, even if one node has 9 and others have 1. TopologySpreadConstraints are quantitative - maxSkew defines acceptable imbalance. With maxSkew 1 and topologyKey zone, if zone-a has 5 web pods and zone-b has 3, the next pod must schedule in zone-b because the skew would otherwise exceed 1. The whenUnsatisfiable field controls behavior: DoNotSchedule blocks pods that would violate maxSkew like required anti-affinity, while ScheduleAnyway allows violations but penalizes them in scoring like preferred anti-affinity. TopologySpreadConstraints are more flexible for controlled spreading - use them when you want even distribution but allow some skew, whereas anti-affinity is for strict isolation like "no two replicas on the same node ever." In practice, topology spread has replaced anti-affinity for most spreading use cases because it prevents the binary failure mode and handles scale-up more gracefully.
