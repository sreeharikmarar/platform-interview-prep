# Talk Tracks

## Q: Explain this in one minute.

**Answer:**

Kubernetes controllers implement a continuous reconciliation loop that makes actual state match desired state. The architecture has three layers: Informers watch the API server and maintain local caches, event handlers extract object keys and enqueue them into rate-limited work queues, and worker goroutines dequeue keys and call the reconcile function. Reconcile reads current state from the cache, computes desired child resources based on the parent's spec, lists actual children, and applies the diff - creating missing resources, deleting extras, and updating mismatches. Critically, reconcile is level-triggered and idempotent - it doesn't care about event history, only current state snapshots. If reconcile returns an error, the work queue requeues the key with exponential backoff. Controllers use owner references to establish parent-child relationships so the garbage collector automatically cleans up children when parents are deleted. This pattern enables self-healing, resilience to restarts, and natural batching of rapid updates.

## Q: Walk me through the internals.

**Answer:**

The Reflector component inside each Informer opens a watch stream to the API server for a specific resource type, starting from a resourceVersion bookmark. As ADDED/MODIFIED/DELETED events arrive, the Reflector updates the Store, which is an in-memory cache keyed by namespace/name with optional custom indexers. The Reflector also runs periodic relist every 10 hours to resync the cache, synthesizing UPDATE events for any differences. Event handlers registered on the Informer receive OnAdd/OnUpdate/OnDelete callbacks. These handlers don't process the object inline - they extract the key and call workqueue.Add(). The work queue deduplicates keys and applies rate limiting. Meanwhile, the controller starts N worker goroutines that each run a loop: call queue.Get() to block until work arrives, invoke reconcile with the key, and call queue.Done() to mark completion. If reconcile returns an error, the worker calls queue.AddRateLimited() which increments a failure counter and requeues with exponential backoff starting at 5ms and capping at roughly 16 minutes. If successful, the worker calls queue.Forget() to reset the failure counter. Reconcile itself reads from the cache using a Lister, computes desired state, fetches actual child resources using label selectors or owner reference queries, diffs them, and issues Create/Update/Delete calls to the API server. Finally, it updates the parent's status subresource with observedGeneration and conditions.

## Q: What can go wrong?

**Answer:**

The most critical failure is reconciliation hot loops - if your reconcile logic isn't idempotent and writes the object even when nothing changed, each write triggers a watch event which enqueues the key again, causing infinite looping. This pegs controller CPU, grows the work queue unbounded, and can OOMKill the pod. The symptom is resourceVersion incrementing rapidly with no apparent changes and workqueue_depth growing unbounded. You prevent this by comparing desired vs actual state before writing and using generation/observedGeneration to skip redundant work. Second, Informer cache staleness causes controllers to make decisions on outdated data - for example, creating duplicate pods because the cache hasn't received the watch event for pods just created. This happens when reconcile runs immediately after a write or when the controller starts before caches sync. The mitigation is waiting for cache sync with WaitForCacheSync before starting workers and handling "already exists" errors gracefully. Third, external side effects without fencing lead to resource leaks - if reconcile calls a cloud API to create a load balancer without checking if it exists, every retry creates a new LB. You fix this by tracking external resource IDs in status, using correlation IDs, and implementing proper cleanup with finalizers.

## Q: How would you debug it?

**Answer:**

I start by checking work queue metrics - workqueue_depth shows how many keys are pending, workqueue_retries_total shows failure rate, and workqueue_longest_running_processor_seconds indicates if workers are stuck. If queue depth is high and growing, I look for hot loops by watching the object with kubectl get -w to see if resourceVersion increments rapidly. I compare metadata.generation to status.observedGeneration - if observedGeneration lags far behind, the controller is failing to reconcile successfully. I dump controller logs with timestamps and grep for the stuck object's key to see reconcile frequency and error messages. For cache staleness issues, I verify the controller waits for cache sync on startup and compare what the controller sees in its cache (via debug logs) with live API state using kubectl get. For external side effects, I check the object's status and annotations for external resource IDs, review controller logs for API call patterns, and audit the external system for orphaned resources. I also use pprof to profile the controller if CPU is high to identify hot code paths, and check API server audit logs to see the controller's request patterns and whether it's being rate limited.

## Q: How would you apply this in a platform engineering context?

**Answer:**

Controllers are the foundation of platform automation. For multi-tenant database provisioning, I build a DatabaseClaim controller that watches DatabaseClaim CRs - when a tenant creates a claim, the Informer enqueues it, reconcile validates the tenant has quota, calls the cloud provider API to provision RDS, and writes status with connection endpoint and observedGeneration. If the cloud API call fails, exponential backoff in the work queue prevents thundering herd retries. For secret rotation, a CertificateRotation controller watches Certificate CRs, checks expiration dates, and when a cert is within 30 days of expiry, enqueues it with RequeueAfter to re-check daily. When rotation is needed, it calls the CA API, creates a new Secret with owner reference to the Certificate, and the garbage collector cleans up old Secrets. For cost optimization, I build a PodAutoscaler controller that watches Pod metrics, computes per-namespace resource usage, and writes NamespaceQuota CRs with recommended limits based on historical usage - this replaces manual spreadsheet-based quota management. The level-triggered pattern is critical for resilience - if the controller crashes for an hour, when it restarts it reads current state and reconciles any missed work without needing event replay. For multi-cluster, a ResourceReplicator controller watches ConfigMaps and Secrets in a hub cluster, and reconciles them to spoke clusters with owner references tracking the hub resource UID, enabling cross-cluster config distribution for platform services.

## Q: Explain level-triggered vs edge-triggered reconciliation.

**Answer:**

Level-triggered means the controller reconciles based on the current state snapshot, not on the event that triggered it. If the desired state is 3 replicas and actual is 2, reconcile creates 1 pod - it doesn't matter whether we got here from deleting 1 pod or creating 2 pods from zero. Edge-triggered would react to specific events like "pod X was deleted, recreate pod X." The advantage of level-triggered is resilience: if the controller crashes and misses 10 delete events, when it restarts it sees 2 pods instead of 12, reconciles to 3, and self-heals. You don't need event replay or persistent queues. Edge-triggered systems need complex event sourcing to handle missed events. Kubernetes chose level-triggered because it simplifies controller logic, makes it idempotent naturally, and enables restarts without state loss. The trade-off is potential inefficiency - you might compute desired state repeatedly even when nothing changed, which is why we use generation/observedGeneration to short-circuit.

## Q: Why use a work queue instead of processing events directly in handlers?

**Answer:**

Work queues provide batching, deduplication, rate limiting, and decoupling. If an object is updated 10 times in 1 second, the queue deduplicates the key so reconcile runs once with the final state rather than 10 times. This is critical for objects with high update frequency like node status. Rate limiting provides exponential backoff for transient failures - if the first reconcile fails due to network timeout, we retry after 5ms, then 10ms, then 20ms up to 16 minutes, preventing tight retry loops from overwhelming the API server. Decoupling event handlers from reconcile allows handlers to be fast and non-blocking - they just extract the key and return, preventing slow reconcile logic from blocking the Informer's event processing loop. Without a queue, if reconcile takes 5 seconds and events arrive every second, you'd fall behind and the Informer buffer would overflow. Finally, work queues enable parallelism - you can run multiple workers pulling from one queue, processing different objects concurrently while still serializing updates to the same object.

## Q: How do controllers handle objects they don't own?

**Answer:**

Controllers typically use label selectors or owner references to filter which objects they reconcile. For example, the Deployment controller watches all ReplicaSets but only reconciles those with ownerReferences pointing to a Deployment with controller: true. If a ReplicaSet has no owner or is owned by a different controller, the Deployment controller ignores it. For objects without owner relationships, controllers use label selectors - the Service controller watches all Pods but only considers those matching the Service's selector. Some controllers also watch objects they don't own but care about for cross-resource logic - for example, the ServiceAccount controller watches both ServiceAccounts and Secrets, creating token Secrets owned by ServiceAccounts. In multi-tenant scenarios, controllers often check namespace labels or RBAC to determine if they should reconcile an object, preventing tenant controllers from mutating objects in namespaces they don't manage. The general pattern is: watch broadly for efficiency (one Informer for all Pods) but filter narrowly in reconcile based on ownership, selectors, or policy.

## Q: How do you implement controller leader election and why is it necessary?

**Answer:**

Leader election ensures only one controller replica reconciles at a time, preventing concurrent reconciliation conflicts. Controllers use the coordination.k8s.io/v1 Lease resource - each replica attempts to acquire a Lease with a specific name, updating it periodically with a heartbeat. The replica that successfully acquires and renews the Lease becomes the leader and starts workers. Non-leader replicas wait and continuously attempt to acquire the Lease in case the leader crashes. Leader election is critical for controllers that manage external resources like provisioning cloud infrastructure - without it, multiple replicas would create duplicate RDS instances or conflict when updating status fields. However, it's not always necessary - controllers that only read state and write status with optimistic locking can run without leader election, providing higher availability since any replica can handle work. I configure leaseD uration based on failover requirements - shorter duration (15 seconds) means faster failover when the leader dies but more API server load from frequent renewals, longer duration (60 seconds) reduces API load but delays failover. For critical platform controllers, I tune renewDeadline and retryPeriod to balance responsiveness and reliability, and monitor the leader election transitions metric to detect flapping which indicates network issues or controller crashes.

## Q: What are controller-runtime's predicates and how do they improve efficiency?

**Answer:**

Predicates in controller-runtime are filter functions that determine whether an event should trigger reconciliation, running before the object key is enqueued in the work queue. The GenerationChangedPredicate triggers when metadata.generation changes between old and new objects, which typically means a spec change. Note: for some resources like Deployments, generation also increments on metadata.annotations changes, not just spec changes — verify generation behavior for your specific resource type. The ResourceVersionChangedPredicate triggers on any change including status, useful for controllers that need to react to status transitions. LabelChangedPredicate and AnnotationChangedPredicate filter events based on label or annotation changes, essential for controllers that watch broadly but only care about labeled objects. Using predicates dramatically improves efficiency - without them, a controller watching Pods would reconcile on every pod status update including heartbeats, overwhelming the work queue. With GenerationChangedPredicate, it only reconciles when pod spec changes like image updates. I implement custom predicates for domain-specific filtering - for example, a certificate renewal controller uses a predicate checking if cert expiration is within 30 days, preventing unnecessary reconciliation of long-lived certificates. Predicates run in the Informer event handler goroutine so they must be fast - complex filtering should happen in reconcile after the event is enqueued. I monitor workqueue_adds_total to verify predicates are effectively reducing reconciliation volume.
