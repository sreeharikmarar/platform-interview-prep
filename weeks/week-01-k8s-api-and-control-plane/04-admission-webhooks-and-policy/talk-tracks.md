# Talk Tracks

## Q: Explain this in one minute.

**Answer:**

Admission control is the synchronous policy enforcement layer in the Kubernetes write path, running after authorization but before persistence to etcd. The admission chain has two phases: mutating admission runs first and can modify objects - built-in plugins inject service account tokens and set defaults, then external mutating webhooks run in parallel to inject sidecars, add labels, or modify fields. The mutated object undergoes schema validation, then validating admission runs second - built-in plugins enforce quotas and limits, ValidatingAdmissionPolicy evaluates CEL expressions, and external validating webhooks can reject but not modify. The API server constructs an AdmissionReview request containing the object, user info, and operation, POSTs it to webhook endpoints over HTTPS, and expects an allowed/denied response with optional JSON patches for mutating webhooks. Webhooks must be fast (under 100ms p99), deterministic (same input produces same output), and highly available because they're on the critical write path - slow or unavailable webhooks block all matching creates and updates cluster-wide. Scope webhooks carefully using objectSelector and namespaceSelector, set appropriate failurePolicy and timeoutSeconds, and use ValidatingAdmissionPolicy for simple validations to avoid external webhook overhead.

## Q: Walk me through the internals.

**Answer:**

When an authenticated and authorized request enters the admission chain, the API server first invokes built-in mutating admission plugins - ServiceAccount admission checks if the pod has a service account specified and injects default if not, then adds automountServiceAccountToken volume and environment variables. DefaultStorageClass admission sets the default storage class on PVCs if unspecified. Then external mutating webhooks run - for each MutatingWebhookConfiguration, the API server evaluates match rules against the request's API group, version, resource, operation, and evaluates objectSelector and namespaceSelector label matches. For matching webhooks, it constructs an AdmissionReview request in JSON containing request.object (the new object), request.oldObject (for updates), request.userInfo, request.operation, and request.dryRun flag. The API server POSTs this to the webhook's clientConfig service or URL over HTTPS, validating the TLS certificate against the provided CA bundle. The webhook responds with an AdmissionReview containing response.allowed (boolean), response.patchType (JSONPatch), and response.patch (base64-encoded array of JSON Patch operations). If allowed and patch provided, the API server applies the patch to the object. All mutating webhook patches accumulate. After mutations, the API server validates the mutated object against its OpenAPI v3 schema. Then validating admission runs - built-in plugins check resource quotas and limits, then ValidatingAdmissionPolicy controllers evaluate CEL expressions like `has(object.metadata.labels.team)`, then external validating webhooks run similarly to mutating but can only return allowed/denied with no patches. If any validator denies, the entire request is rejected with the denial message. If all pass, the object is persisted to etcd.

## Q: What can go wrong?

**Answer:**

The highest-impact failure is webhook unavailability causing cluster-wide write blockage. If a webhook pod crashes or is unreachable and has failurePolicy Fail with broad match rules like all resources in all namespaces, every kubectl apply hangs for timeout seconds then fails. This blocks not just user operations but all controller reconciliation including status updates, GitOps systems, and critical incident response. I've seen production outages where a misconfigured istio-sidecar-injector webhook matching all pods prevented any pod creation across the cluster. The mitigation is using narrow scope with objectSelector matching only namespaces with inject-sidecar label, excluding kube-system, setting low timeoutSeconds like 5 seconds, deploying webhooks with high availability and PodDisruptionBudget, and starting with failurePolicy Ignore during rollout. Second, non-deterministic mutation causes continuous apply churn where the same manifest produces different outputs on each apply. If a webhook injects a timestamp annotation or pulls latest from an external config source, every GitOps reconcile detects drift and reapplies, creating API server load and making it impossible to detect true configuration changes. Third, policy conflicts between webhooks where webhook A sets image to one value and webhook B overwrites it, leading to unpredictable final state depending on webhook execution order. Unlike SSA field managers, webhooks don't have explicit conflict detection, so you coordinate via objectSelector scope boundaries.

## Q: How would you debug it?

**Answer:**

For webhook availability issues, I start by listing all webhook configurations with kubectl get validatingwebhookconfigurations,mutatingwebhookconfigurations and inspecting their match rules, namespaceSelector, objectSelector, failurePolicy, and timeoutSeconds. I check if the webhook service and endpoints exist and if pods are running with kubectl get svc,endpoints,pods in the webhook namespace. I look at API server logs for webhook call failures, timeout errors, and latency spikes with kubectl logs kube-apiserver grepping for the webhook name. I test the webhook endpoint directly using curl from a test pod to verify network policy isn't blocking traffic. For non-deterministic mutations, I apply the same manifest twice with --dry-run=server and diff the outputs to see if the mutated object differs. I inspect managedFields to see if the webhook's fieldManager shows changing values across applies. I watch the object with kubectl get -w to observe resourceVersion incrementing on every reconcile. For policy conflicts, I list all webhooks matching the affected resource type and compare their match rules to identify overlap. I check API server audit logs to see webhook execution order for a specific request. I test webhooks in isolation by temporarily disabling others to understand each webhook's mutations independently.

## Q: How would you apply this in a platform engineering context?

**Answer:**

Admission webhooks are essential for enforcing platform-wide policies and injecting infrastructure requirements. For multi-tenancy, I implement a validating webhook that enforces ResourceQuota boundaries - when a tenant creates a Deployment, the webhook calculates total requested resources across their namespace and rejects if it exceeds their quota, providing immediate feedback instead of waiting for the ResourceQuota controller to fail pod creation. For security, a validating webhook enforces that production namespaces can't use latest image tags, can't run privileged containers, and must have resource limits set, preventing resource exhaustion and security vulnerabilities. For sidecar injection, a mutating webhook matching labeled namespaces injects monitoring agents, log shippers, and security scanners into every pod, ensuring consistent observability without requiring tenant teams to manually configure these. The webhook sets owner references so when the pod is deleted, injected config is cleaned up. For cost allocation, a mutating webhook injects billing labels (tenant, team, cost-center) that map to namespace labels, and sets these as immutable so tenants can't manipulate chargeback data. I use ValidatingAdmissionPolicy for simple structural validation like requiring all ConfigMaps to have an owner annotation, avoiding external webhook overhead. For complex validation like checking if a referenced Secret exists in a different namespace, I use external webhooks that query the API server with proper RBAC.

## Q: When should you use a webhook vs a controller vs data-plane enforcement?

**Answer:**

Use admission webhooks for synchronous validation and mutation that must happen before the object is persisted - things like "no latest image tags in prod" or "inject sidecar proxy." Webhooks are on the critical write path, so they must be fast and highly available. Use controllers for asynchronous reconciliation that doesn't block the write - for example, a controller that watches Services and creates corresponding DNS records, or computes xDS snapshots from Gateway config. Controllers can take seconds or minutes, handle external dependencies with retries, and don't impact user-facing latency. Use data-plane enforcement for runtime policies that change based on actual traffic - for example, Envoy's rate limiting, circuit breaking, or JWT validation. The data plane sees live requests and can make decisions with sub-millisecond latency that would be impossible in the control plane. A concrete example: for mutual TLS enforcement, use an admission webhook to validate that workloads have the required sidecar annotations (fast validation at create time), use a controller to watch workloads and compute SPIFFE identities, then use Envoy's RBAC filter to enforce mTLS at runtime (actual traffic inspection).

## Q: How do ValidatingAdmissionPolicy and CEL compare to external webhooks?

**Answer:**

ValidatingAdmissionPolicy with CEL expressions runs in-process within the API server, avoiding external HTTP calls, TLS certificate management, and webhook availability concerns. Policies are declared as Kubernetes resources and evaluated by built-in controllers, giving sub-millisecond latency compared to 10-100ms for webhook round trips. CEL is type-safe and validated at policy creation time, catching syntax errors before they affect production. However, CEL is less expressive than full programming languages - you can't call external APIs, query databases, or implement complex business logic. For example, "all deployments must have a team label" is trivial in CEL: `has(object.metadata.labels.team)`, but "verify the team label matches an entry in LDAP" requires an external webhook. ValidatingAdmissionPolicy is perfect for simple structural validation, label requirements, and field constraints. Use external webhooks for complex validation needing external state, cross-resource checks across namespaces, or integration with external systems. In multi-tenant platforms, I use ValidatingAdmissionPolicy for quick checks like "DatabaseClaim must specify retention days" and external webhooks for authorization checks against the platform's RBAC service or tenant quota validation that requires querying current resource usage across namespaces.

## Q: Explain the failure policy tradeoff.

**Answer:**

FailurePolicy Fail prioritizes security over availability - if the webhook is unreachable or times out, the request is rejected. This prevents the system from entering an invalid state when policy enforcement is down, but creates availability risk if the webhook has operational issues. Use Fail for critical security boundaries like "no privileged pods in prod" or "all images must come from approved registries" where violating the policy is worse than blocking deployments. FailurePolicy Ignore prioritizes availability - if the webhook fails, the request proceeds as if the webhook passed. This keeps the cluster operational during webhook outages but creates security risk if the policy is critical. Use Ignore for non-essential mutations like "add monitoring annotations" or during initial webhook rollout before stability is proven. The best practice is start with Ignore and comprehensive testing, monitor webhook availability and latency for a week, then switch to Fail once you're confident. Also scope webhooks narrowly so a Fail policy affects only specific namespaces, reducing blast radius. For truly critical policies, consider redundancy - run multiple webhook replicas across availability zones with aggressive PodDisruptionBudget and anti-affinity, plus circuit breaker logic in the webhook to fail open if external dependencies are down.

## Q: How do you implement async validation patterns with admission webhooks?

**Answer:**

For validations that take too long for synchronous admission (image scanning, external policy engines), I use a two-phase approach. First, a mutating webhook injects a pending status annotation and allows the object through quickly. Second, an async controller watches for objects with pending status, performs the slow validation (calls image scanner API, queries policy engine), and either updates the object's status with approval or deletes it if validation fails. For Pods, I use a validating webhook that checks status.conditions for an approved condition - initially the condition is missing so the webhook rejects, the async controller adds the condition after validation completes, and subsequent pod recreation (by deployment controller) passes validation. This pattern keeps the admission path fast while still enforcing complex policies. The tradeoff is eventual enforcement - there's a window where invalid objects exist. I mitigate this by using OPA Gatekeeper's audit mode which continuously scans existing resources and reports violations, and by implementing finalizers that block deletion until the async validation completes, preventing zombie resources. For critical security policies, I still use synchronous webhooks with tight SLOs and fallback to cached validation results when the external system is slow.

## Q: How do you version and evolve admission webhook schemas safely?

**Answer:**

Admission webhooks must handle multiple API versions because old controller versions might send v1beta1 while new ones send v1. I implement version negotiation by declaring all supported versions in the MutatingWebhookConfiguration admissionReviewVersions field (e.g., ["v1", "v1beta1"]). The webhook inspects request.apiVersion and responds with the same version. For the object being admitted, I use server-side conversion by setting conversion strategy in the CRD - the API server converts requests to the webhook's preferred storage version. When evolving validation logic, I deploy new webhook versions with backward-compatible rules first, monitor for rejections in lower environments, then progressively tighten enforcement. For breaking changes, I use feature gates - the webhook checks a ConfigMap or annotation to enable new validation rules per namespace, allowing gradual rollout. I version the webhook itself using Deployment rolling updates with maxUnavailable=0 to maintain availability during upgrades. For schema changes in the AdmissionReview, I test against all supported Kubernetes versions in CI using kubebuilder envtest. I monitor webhook request versions using custom metrics to know when old versions are no longer in use and can be deprecated.
