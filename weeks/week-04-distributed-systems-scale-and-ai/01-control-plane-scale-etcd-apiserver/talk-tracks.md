# Talk Tracks

## Q: Explain control plane scaling in one minute.

**Answer:**

Control plane scaling comes down to three bottlenecks and three axes. The bottlenecks are: etcd write throughput, because all state mutations serialize through Raft consensus and a single etcd cluster tops out around 10,000 writes per second under ideal conditions; watch fan-out, because every write to etcd triggers notifications to potentially thousands of watchers, and the API server's watch cache must fan out those events under memory and CPU pressure; and admission latency, because every mutating or validating webhook call is synchronous in the write path, so a webhook with 200ms p99 latency adds directly to API server write throughput limits. The three scaling axes are vertical — giving the API server more CPU for request fan-out and etcd more IOPS for write commit latency; horizontal — running three to five API server replicas behind a load balancer, which scales read throughput but not write throughput since all writes still serialize through etcd; and architectural — sharding controllers by namespace or resource group, separating events into a dedicated etcd cluster to remove their write noise from the critical path, and coalescing status updates to reduce write amplification. A large cluster handling 5,000 nodes and 100,000 pods will be bottlenecked primarily by etcd write throughput and watch fan-out before it is ever bottlenecked by API server CPU.

## Q: Walk me through what happens to etcd when 1,000 pods update status simultaneously.

**Answer:**

When 1,000 pods update their status simultaneously, the API server receives 1,000 PATCH or PUT requests for Pod objects. Each request goes through the full admission chain, then hits etcd as a serialized write. etcd uses the Raft consensus protocol, so each write must be proposed by the leader, acknowledged by a quorum of followers, and then committed before the API server gets a response. Under this burst, the etcd leader's commit queue grows, and commit latency climbs — you will see `etcd_disk_backend_commit_duration_seconds` p99 rise from a baseline of 5-10ms toward 50-100ms or higher depending on disk IOPS. Each of those 1,000 writes also increments the global cluster revision, which is etcd's monotonic counter for all objects. The watch cache on the API server maintains a circular buffer of recent events indexed by revision; under a 1,000-write burst, that buffer fills quickly and slow watchers fall behind. The write amplification is significant: a Pod status write that is 4KB serialized as protobuf gets written to etcd's WAL, then to the B-tree backend, then replicated over the Raft log to each follower, and then fanned out to every watcher of Pods across all controller replicas. Compaction pressure also increases because each revision stores the full new value of the object — 1,000 Pod writes generate 1,000 new key-value revisions in the B-tree, and without periodic compaction via `--auto-compaction-retention`, the database grows unbounded. After the compaction removes old revisions, the freed pages are not returned to the OS until you run defragmentation, so physical disk usage remains high even after compaction.

## Q: What can go wrong at scale?

**Answer:**

Three high-blast-radius failure modes stand out at large scale. First, etcd quota exceeded: etcd enforces a default storage limit of 8GB controlled by `--quota-backend-bytes`. If compaction is not tuned or if a controller produces excessive churn — for example, a buggy operator that updates an annotation on every reconcile loop — the database grows to quota and etcd enters a read-only alarm state. The blast radius is total: all API server writes fail with a 429 `etcdserver: mvcc: database space exceeded` error, meaning no pods can be scheduled, no deployments can roll out, and no stateful operations can complete until the alarm is cleared by running `etcdctl defrag` and `etcdctl alarm disarm`. Second, relist storms: when the API server restarts or the watch cache is reset, every controller's Informer receives a 410 Gone on its watch and falls back to a full List operation. In a cluster with 100,000 pods across 500 controller replicas, simultaneous relists generate a request spike that can overwhelm the API server's in-flight limit and trigger APF rejection of new requests. Third, API Priority and Fairness starvation: if tenant workloads are classified into the same PriorityLevel as system controllers — for example, because a custom FlowSchema is missing — a burst of `kubectl exec` or `kubectl logs` requests from hundreds of developers can exhaust concurrency shares and starve the `node-lifecycle` or `endpoint-slice` controllers, delaying pod health propagation cluster-wide. Each of these has a distinct detection path via metrics and distinct mitigation strategies.

## Q: How would you debug slow API server responses?

**Answer:**

I start with the APF metrics because starvation is a common and invisible cause of slow responses. `apiserver_flowcontrol_request_wait_duration_seconds` broken down by `priority_level` and `flow_schema` shows which priority levels are queueing. If `workload-high` or `global-default` have p99 queue wait above 100ms, it indicates concurrency exhaustion for those levels. Next I check `apiserver_request_duration_seconds` by verb and resource — if `get pods` p99 is 2 seconds but `get nodes` p99 is 50ms, the issue is localized to the Pod watch cache or a Pod-specific admission webhook. For etcd-specific latency I look at `etcd_request_duration_seconds` by operation type: `txn` for writes, `range` for reads. If `txn` p99 is above 25ms, I check `etcd_disk_backend_commit_duration_seconds` for disk IOPS exhaustion and `etcd_network_peer_round_trip_time_seconds` for Raft replication latency between members. For admission webhooks I check `apiserver_admission_webhook_request_total` and `apiserver_admission_webhook_admission_duration_seconds` — a slow webhook shows up immediately as a write latency outlier. I also check `apiserver_watch_cache_capacity_overflow_total` and `watch_cache_capacity_adjustment_total` to see if the watch cache is being resized under pressure, which causes temporary stalls. Finally, I correlate all of this with the API server's goroutine count via `/metrics` or pprof — if goroutines exceed 10,000, the API server may be under a goroutine leak from hung watch connections.

## Q: How would you apply this in a platform engineering context?

**Answer:**

In platform engineering, control plane scaling decisions directly affect multi-tenant reliability. For etcd capacity planning, I size the etcd cluster based on expected object count and churn rate, not just node count — a cluster running a service mesh operator that writes Envoy config as ConfigMaps at high frequency needs more careful etcd IOPS budgeting than a static workload cluster of the same node count. I use `etcdctl endpoint status` and the `etcd_mvcc_db_total_size_in_bytes` metric to track growth trends and set alerting at 60% of quota to give the team time to defrag before an outage. For multi-tenant API server tuning, I create custom FlowSchema resources that give platform controllers — ArgoCD application controllers, cert-manager, external-secrets — dedicated PriorityLevel concurrency shares isolated from tenant `kubectl` traffic. This prevents a developer running `kubectl exec -it` in a tight loop from degrading the reconciliation throughput of infrastructure controllers. For controller sharding, when we run a namespace provisioning operator that must reconcile 10,000 namespaces, I partition reconciliation by namespace hash modulo N shards, with each shard holding a distributed lock via a Kubernetes Lease object. This keeps per-controller queue depth bounded regardless of cluster growth. For etcd capacity events, I have runbooks that automate the defrag workflow: cordon writes by scaling the operator to zero, run `etcdctl defrag --cluster`, verify database size dropped, then disarm the quota alarm and scale back up — all as a structured incident response script so the on-call engineer does not have to improvise under pressure.

## Q: How does API Priority and Fairness work and why does it matter?

**Answer:**

API Priority and Fairness, enabled by default since Kubernetes 1.20, replaces the flat `--max-requests-inflight` and `--max-mutating-requests-inflight` limits with a multi-level queuing and scheduling system. The core objects are FlowSchema and PriorityLevel. Every incoming request is matched against FlowSchema resources in order of `matchingPrecedence` — lower number means higher priority. A FlowSchema defines a `distinguisherMethod` (by user or by namespace) which determines how requests are bucketed into flows within the priority level. The PriorityLevel resource defines a `nominalConcurrencyShares` value; the API server computes each level's actual concurrency limit proportionally to its shares relative to the total shares across all levels. Requests that exceed their level's concurrency limit are queued using a shuffle-sharding algorithm: each flow is hashed to a small subset of queues, and the queue with the shortest depth is used. This prevents a single misbehaving flow from monopolizing all queue slots within a priority level. It matters because without it a burst of low-priority requests — `kubectl get pods --watch` from a monitoring tool or a CI pipeline hammering `kubectl apply` — can consume all API server concurrency and starve system controllers like `kube-controller-manager` and `kube-scheduler`. The exempt PriorityLevel exists for health checks and leader election to ensure they are never queued. I monitor `apiserver_flowcontrol_rejected_requests_total` by `reason` (queue-full vs timeout) and `apiserver_flowcontrol_current_inqueue_requests` to detect when any level is chronically saturated and needs its concurrency share increased or the workload causing saturation needs to be rate-limited upstream.

## Q: What is the difference between etcd compaction and defragmentation?

**Answer:**

Compaction and defragmentation address two different aspects of etcd storage management and are often confused because both are run to reclaim space. Compaction operates at the MVCC layer: etcd stores every revision of every key so that watch clients can replay history. `etcdctl compact <revision>` removes all historical key revisions below the specified revision from the B-tree index, making those old revisions inaccessible to watch queries. After compaction, `etcdctl endpoint status` will show a smaller `dbSize` number, but only in terms of logical pages — the freed pages are marked as free in the BoltDB freelist but are not returned to the operating system. The physical file on disk does not shrink. Defragmentation operates at the BoltDB file level: `etcdctl defrag --cluster` rewrites the BoltDB database file from scratch, placing all live pages contiguously and returning unused pages to the OS. After defrag, the physical file size drops to match the logical used size. The critical operational difference is timing and risk: compaction can run online against a live cluster with no interruption and should run periodically — typically controlled by the `--auto-compaction-retention` flag set to `"5m"` or `"1000"` revisions. Defragmentation is more disruptive: it locks the etcd member during the rewrite, causing a brief unavailability for that member, so it must be run one member at a time and during a maintenance window. Running defrag on the leader causes a leader election. The operational sequence for a quota alarm is: compact first to free logical space, then defrag one member at a time to reclaim physical space, then disarm the quota alarm with `etcdctl alarm disarm`.

## Q: How would you design controller sharding for a 10,000-namespace cluster?

**Answer:**

For a controller that must reconcile objects across 10,000 namespaces, a single-instance controller will have a work queue that grows proportionally with namespace count and event churn, creating unbounded reconciliation latency. The sharding approach I use is namespace-hash-based partitioning with Kubernetes Lease objects for distributed coordination. Each controller replica computes `hash(namespace) mod N` where N is the number of shards, and only enqueues namespaces whose hash matches its assigned shard index. Shard assignment is stored in a Kubernetes Lease resource — one Lease per shard — and each controller replica holds a single lease via the `controller-runtime` lease manager or a custom leader-election loop using the `k8s.io/client-go/tools/leaderelection` package. The Informer for the watched resource type runs on every replica to keep the watch cache warm, but the Reconcile function gates processing on shard ownership before doing any work. For rebalancing when a replica crashes: the dead replica's lease expires after `leaseDurationSeconds`, and a surviving replica can acquire it, taking on that shard's namespaces. Consistent hashing avoids the rebalancing problem of modulo-N sharding when shard count changes — using a ring-based algorithm, adding one new shard only moves roughly `total_namespaces / (N+1)` namespaces rather than rehashing everything. For correctness, each reconcile must still handle the case where it processes a namespace it no longer owns due to a lease transition race, which means the reconcile must re-check lease ownership before committing side effects. Metrics to monitor are per-shard queue depth via `workqueue_depth` labeled by shard index, and lease renewal failures via `leaderelection_leader_transitions_total`.

## Q: When would you run separate etcd clusters and what are the trade-offs?

**Answer:**

The canonical use case for a separate etcd cluster is Kubernetes Events. Events are extremely high-write-volume objects — every scheduler decision, every kubelet state transition, every controller action writes an Event — and their loss is operationally tolerable because they are informational, not stateful. By configuring the API server with `--etcd-servers-overrides='/events#http://etcd-events:2379'`, all Event writes go to a dedicated etcd cluster, removing them from the critical-path etcd cluster entirely. This can reduce write load on the primary etcd by 30-50% in high-churn clusters. A second use case is separating by API group: if you have a high-frequency CRD — for example, a telemetry ingestion CR that receives millions of writes per day — you can route writes for that specific group to a dedicated etcd cluster, giving it its own IOPS budget without competing with core Kubernetes resources. The trade-offs are significant: each separate etcd cluster requires its own quorum of three to five nodes, its own backup and DR procedures, its own monitoring and alerting, and its own defrag and compaction schedules. Operational complexity scales with the number of clusters. Debugging becomes harder because a failing etcd cluster for a non-obvious resource group causes partial API server failures that look like intermittent errors rather than a clean outage. You also lose the ability to do atomic cross-resource transactions between etcd clusters, which matters if your CRD has strong consistency dependencies on built-in resources. My recommendation is to use the events etcd split universally for clusters above 1,000 nodes, and only consider further splits for specific high-volume CRDs when write contention is measurable in etcd metrics rather than as a preemptive architectural choice.
