# Talk Tracks

## Q: Explain overload protection in one minute.

**Answer:**

Overload protection works in three layers, each covering what the previous one misses. The first layer is rate limiting at the gateway: you bound the number of requests per second a tenant can submit, so a single caller cannot consume unbounded capacity. The second layer is circuit breaking at the connection pool: you set hard limits on `max_connections`, `max_pending_requests`, and `max_requests` per upstream cluster, so a slow or degraded downstream cannot cause connection exhaustion in your application. The third layer is adaptive load shedding inside the service itself: when CPU or queue depth exceeds a threshold the server starts rejecting requests with 503 before accepting them, trading individual request failures for overall system stability. The most dangerous failure mode is not hard-crash failures — it is gray failures, where a service is still accepting connections but processing them one hundred times slower than normal. Timeouts are the only defense against gray failures, and they must be set end-to-end: gateway timeout, client timeout, and server-side request deadline all need to be consistent. Retry budgets close the amplification loop: a per-request retry limit of two looks safe in isolation, but across a fleet of a thousand clients retrying simultaneously you turn a 10% error rate into a 30% load spike. A retry budget expressed as a percentage of total in-flight requests — typically 20% — bounds retries system-wide regardless of how many individual retries seem locally reasonable.

## Q: Walk me through how a retry storm develops.

**Answer:**

Start with steady state: 1,000 requests per second flowing through a gateway to a backend cluster of 10 pods, each handling 100 RPS with 20 ms p50 latency. One pod develops a garbage-collection pause and its latency climbs to 2,000 ms. The gateway has a timeout of 500 ms. After 500 ms, the gateway times out 100 requests per second — the share that was hitting the slow pod — and retries each one once. The cluster now receives 1,100 RPS: 900 normal plus 100 retried. The 100 retried requests land on the remaining nine pods, increasing their load by about 11 RPS each, pushing them from 100 RPS to 111 RPS. If those pods are near their capacity knee, their latency starts climbing. Now more requests time out — say 200 per second — and retries jump to 200 per second, pushing load to 1,200 RPS. This is the amplification loop. Within three to five timeout intervals — 1.5 to 2.5 seconds in this example — every pod is overloaded and the entire cluster is returning 503s for every request even though the original cause was a single pod's GC pause. The load on the cluster at the point of total failure is 3x to 5x the original load. Recovery is blocked because every client, seeing failures, retries as fast as its timeout allows, keeping the cluster saturated. The system is now metastable: it cannot recover without external intervention such as shedding all load for a period longer than the retry window. The fix requires three simultaneous controls: a retry budget that caps retries at 20% of traffic so 1,000 RPS cannot become more than 1,200 RPS, exponential backoff with jitter so retries spread over time, and a circuit breaker that opens when the error rate exceeds a threshold and stops retries entirely until the backend has had time to drain.

## Q: What can go wrong with circuit breakers?

**Answer:**

Circuit breakers fail in two opposite directions, and both are costly. Too tight — `max_connections: 5`, `max_pending_requests: 2` on a cluster that routinely needs 20 concurrent connections at p95 — and the breaker trips during normal traffic spikes. Callers see 503s from Envoy's `upstream_rq_pending_overflow` counter with response flag `UO` (upstream overflow) even though the backend is healthy. Engineers see the errors, assume backend problems, restart pods, and the circuit breaker threshold is never revisited. This creates false 503s that erode SLO budgets and train teams to ignore overflow metrics. Too loose — `max_connections: 10000`, `max_requests: 50000` — and the circuit breaker provides no protection: a slow backend can absorb all connection slots across a fleet before a single overflow is recorded. The second problem is interaction between circuit breaking and outlier detection. Envoy's outlier detection (`consecutive_5xx`, `consecutive_gateway_failure`, `success_rate`) ejects individual unhealthy hosts from the load balancer, while circuit breaking applies aggregate limits to the cluster as a whole. When a backend pod starts throwing 5xx errors, outlier detection ejects it. That concentrates load on remaining pods, which may now trip the aggregate circuit breaker, generating `UO` overflows on the whole cluster even though the breaker stats look like an aggregate capacity problem rather than a single host problem. Reading `cluster.my_service.outlier_detection.ejections_active` alongside `upstream_rq_pending_overflow` is necessary to tell these two mechanisms apart. A third failure mode is state mismatch across multiple Envoy replicas: each Envoy in a fleet maintains its own circuit breaker state independently. A backend that is overloaded for one Envoy instance may be healthy from another's perspective, so aggregate overflow stats in Prometheus can look fine while individual Envoy pods are shedding load. Always check per-pod stats from individual admin endpoints, not just the aggregated Prometheus scrape.

## Q: How would you debug a cascading 503 event?

**Answer:**

Start at the access log and read the response flags field immediately — do not guess at causes from alert names. A `UO` flag (upstream overflow) means the circuit breaker tripped and Envoy rejected the request without touching the backend. A `UH` flag (no healthy upstream) means outlier detection ejected all endpoints. A `UF` flag (upstream connection failure) means TCP connections to backends are failing. Each flag points to a different root cause layer.

For `UO`, go to the Envoy admin interface: `curl -s http://localhost:9901/stats | grep upstream_rq_pending_overflow`. A steadily incrementing counter confirms the breaker is tripping. Then check the circuit breaker thresholds with `curl -s http://localhost:9901/clusters | grep -A 20 "my_service"` and look at `cx_active` alongside `max_connections`. If `cx_active` equals `max_connections`, the pool is saturated.

For `UH`, run `curl -s http://localhost:9901/clusters | grep -E "health_flags|ejected"` and look for `failed_active_hc` or `ejected_via_outlier_detection`. Then cross-reference with `curl -s http://localhost:9901/stats | grep "outlier_detection.ejections_enforced_consecutive_5xx"` — a non-zero and incrementing value tells you the backend pods were throwing 5xx errors that triggered ejection.

For queue depth, query `upstream_rq_pending_total` and `upstream_rq_pending_active` to see whether requests are accumulating. If pending requests are rising while `cx_active` is flat, the backend is accepting connections but not draining them — classic slow-consumer symptom.

Distributed tracing closes the loop: find a representative failing trace ID from the access log's `x-request-id` field, pull the trace in Jaeger or Tempo, and look at the span timing. If the upstream span is missing entirely, Envoy never forwarded the request. If the upstream span exists but exceeds the timeout value, the timeout is too tight relative to backend p99 latency. If the upstream span shows a 5xx status code, the problem is in the backend, not the proxy layer.

## Q: How would you apply this in a platform engineering context?

**Answer:**

In a shared API gateway serving multiple internal teams, rate limiting is the primary mechanism for preventing noisy-neighbor incidents. The pattern that works at scale is tenant-aware rate limiting using Envoy's global rate limit service (RLS): the gateway forwards a descriptor containing the tenant identifier — extracted from a JWT claim or a header like `X-Tenant-ID` — to a central RLS backed by Redis, which applies per-tenant token bucket limits and returns OVER_LIMIT when the budget is exhausted. This means a single tenant bursting to 10x their allocation triggers `RL` response flags only for their own traffic; other tenants are unaffected. The key operational discipline is instrumenting `ratelimit_service.total_hits` and `over_limit` counters per descriptor so you can give tenants real usage data and right-size their quotas instead of setting arbitrary limits. For shared infrastructure like etcd, PostgreSQL, or an internal secrets manager, circuit breaking at the gateway level protects the shared resource from any single consumer taking it down. You set `max_connections` on the upstream cluster to a value below the resource's hard connection limit, configure outlier detection with `consecutive_gateway_failure: 3` and `base_ejection_time: 30s`, and add a retry budget of 20% so that client retries during a partial outage do not push the resource over its connection limit. The outcome is that a single team's poorly configured client flooding the secrets manager with requests hits its circuit breaker threshold, starts receiving 503s locally, and the rest of the platform continues operating. Pair this with per-tenant rate limit dashboards and automated alerts on `upstream_rq_pending_overflow` counts that page the tenant's team, not the platform team, and you shift operational burden appropriately.

## Q: What is the difference between circuit breaking and outlier detection in Envoy?

**Answer:**

Circuit breaking and outlier detection solve different problems and operate at different scopes, but they interact and must be tuned together. Circuit breaking in Envoy applies aggregate limits to a cluster as a whole. The `circuit_breakers.thresholds` block sets `max_connections` (active TCP connections to all endpoints combined), `max_pending_requests` (requests queued waiting for a connection), `max_requests` (in-flight HTTP requests), and `max_retries` (concurrent retry attempts). When any of these counters hits its ceiling, Envoy immediately rejects new requests with a 503 and sets the `UO` response flag; dedicated overflow counters exist for `upstream_rq_pending_overflow` and `upstream_cx_overflow`. The circuit breaker does not care which specific endpoints are slow or unhealthy — it sees only aggregate saturation. Outlier detection, configured in the `outlier_detection` block on the cluster, works at the individual endpoint level. It tracks error rates per backend pod — `consecutive_5xx`, `consecutive_gateway_failure`, `success_rate` relative to the cluster average — and ejects misbehaving endpoints from the load balancer for a duration equal to `base_ejection_time` multiplied by the ejection count (default 30 seconds, growing linearly: 30s, 60s, 90s, etc.). An ejected endpoint still counts toward `membership_total` but not `membership_healthy`. Circuit breaking then sees a smaller healthy pool and may trip sooner because the remaining endpoints are carrying more load. Both mechanisms are necessary: outlier detection removes individually sick pods from rotation without taking down the whole cluster; circuit breaking prevents a completely saturated upstream from consuming all connection resources in the calling service. Omitting outlier detection means one bad pod absorbs its fair share of traffic indefinitely. Omitting circuit breaking means a total backend failure can exhaust caller-side connection pools across your entire fleet.

## Q: Explain retry budgets and why per-request retry limits are insufficient.

**Answer:**

A per-request retry limit of two looks conservative in isolation. A single request that fails once retries once — a 2x amplification that seems manageable. The problem is that this reasoning ignores fleet-level behavior. Consider a gateway handling 1,000 RPS with a 10% error rate — 100 errors per second. With per-request retry limit of two, those 100 failed requests each trigger up to two retries, adding up to 200 extra requests per second to the backend. The backend is now receiving 1,200 RPS instead of 1,000 RPS. If the error rate is caused by backend overload, the extra 200 RPS makes it worse. Now the error rate climbs to 20%, generating 200 errors per second, each retried twice — 400 extra RPS — and the load climbs to 1,400 RPS. This is the retry amplification loop. A retry budget breaks the loop by expressing retries as a percentage of total in-flight requests rather than a per-request count. Envoy's `retry_budget` block sets `budget_percent` (default 20%) and `min_retry_concurrency` (default 3). With a 20% budget on a 1,000 RPS gateway, the maximum additional load from retries is capped at 200 RPS regardless of how many individual requests fail. When the budget is exhausted, Envoy stops issuing retries cluster-wide and returns the original error to the caller immediately. The math changes: 1,000 RPS + 20% = 1,200 RPS maximum load on the backend. That is a bounded and plannable load ceiling rather than an unbounded feedback loop. The `upstream_rq_retry_overflow` stat in Envoy tracks how many retries were suppressed by the budget — a non-zero and sustained value is your signal that retries are being actively throttled, which tells you the backend is degraded and needs attention rather than more retries.

## Q: How does Little's Law help you size rate limits?

**Answer:**

Little's Law states L = lambda * W, where L is the mean number of requests in the system (concurrency), lambda is the arrival rate (requests per second), and W is the mean time each request spends in the system (latency in seconds). The law applies to any stable system in equilibrium and lets you derive any one of the three values if you know the other two. For rate limit sizing, the most useful form is: maximum safe arrival rate = maximum concurrency / target latency ceiling. Worked example: your backend service can handle 50 concurrent requests before latency degrades significantly — you determined this from load testing where p99 latency stayed below 200 ms up to 50 concurrency and jumped to 2,000 ms at 60 concurrency. Your SLO target latency is 200 ms = 0.2 seconds. Plugging into Little's Law: lambda = L / W = 50 / 0.2 = 250 RPS. That is your rate limit ceiling — at arrival rates above 250 RPS the system will exceed 50 concurrent requests and latency will blow out. Set your gateway rate limit at 250 RPS per cluster, with per-tenant limits sized proportionally to their fair share of capacity. The second use of Little's Law is sizing connection pools for Envoy's circuit breaker `max_connections`. If your backend handles 200 RPS at a mean latency of 50 ms = 0.05 seconds, the expected concurrent connections are 200 * 0.05 = 10. Set `max_connections` to 15 to 20 — 1.5x to 2x the steady-state value — to absorb normal spikes without tripping the breaker under healthy conditions. The law gives you a principled starting point; you still need to validate under real traffic patterns because actual request latency distributions are skewed and the heavy tail matters more than the mean for connection pool sizing.

## Q: Describe a metastable failure you have seen or studied.

**Answer:**

A metastable failure is a state where the system has moved away from its stable operating point and cannot return to it without external intervention, even after the original trigger has resolved. The defining characteristic is that the recovery mechanism itself becomes a source of additional load that prevents recovery. The thundering herd pattern is the canonical example. Consider a caching layer in front of a database. The cache holds 10,000 keys, each with a TTL of five minutes. Under normal operation, keys expire gradually — roughly 33 per second — and each cache miss generates one database query, easily handled. Now a deployment flushes the entire cache at once. All 10,000 keys expire simultaneously. Every request to the cache misses, every caller queries the database directly, and the database — sized for 33 queries per second of miss traffic, not 1,000 queries per second — becomes fully saturated. Database query latency climbs from 5 ms to 2,000 ms. Callers time out and retry. Retries generate additional database queries. The database cannot drain because new queries are arriving faster than it can process them. Even if the cache is repopulated, it cannot be repopulated faster than the database can serve queries, which is slower than the query arrival rate. The system is stuck: the only way out is to stop all traffic to the database — shedding load entirely — for long enough that the database drains its queue, then gradually reintroduce traffic while the cache warms up. The Kubernetes-specific variant is a CrashLoopBackoff cascade: a critical shared service crashes, every dependent pod restarts and immediately makes initialization calls to the shared service, and the shared service is overloaded before it can finish its own startup sequence. The fix requires exponential backoff with jitter on initialization retries, readiness gates that prevent downstream pods from starting until the shared service declares itself ready, and a circuit breaker on the initialization path so that a partially started system does not generate its own retry storm before the first pod is healthy.
