# Talk Tracks

## Q: Explain inference gateways in one minute.

**Answer:**

Inference gateways solve a problem that general-purpose HTTP proxies were not designed for: request cost is non-uniform. In a normal microservice, every POST to `/api/order` has roughly the same cost — a few database reads, some logic, a response. In an inference system, one request generates 40 tokens in 80ms and another generates 4,000 tokens in 8 seconds. Both are POST `/v1/chat/completions`. If you rate-limit by request rate, you have no control over actual GPU utilization, because a user who sends one request per minute with a 4,000-token completion can still pin a GPU for minutes at a time. An inference gateway must understand tokens, not requests. The critical capabilities that differentiate an inference gateway from a generic reverse proxy are: model-version routing (routing based on headers like `x-model` or `model` in the request body, not just URL path), token-budget enforcement (decrement a per-tenant token counter after each response, block when depleted), queue-aware load balancing (route to the backend GPU pod with the shortest pending queue rather than round-robin, because queue depth is the leading indicator of time-to-first-token), streaming-aware metrics (TTFT and tokens-per-second require intercepting the SSE response stream, not just recording response time), and canary routing (split traffic by weight across model versions with independent rollback). The core insight is that GPU time is the scarce resource and tokens are its unit of measure, not requests.

## Q: Walk me through the request path for an inference request.

**Answer:**

The critical path for an inference request through a production inference gateway has eight stages, each of which adds latency if it stalls. First, TLS termination: the client connection is terminated at the gateway's listener, certificates are validated, and the downstream connection is upgraded to HTTP/2 to support streaming. Second, model header extraction: the gateway reads the `model` field from the JSON request body or an `x-model-id` header to identify which model version is being requested — this is necessary before any routing decision can be made. Third, tenant identification: the gateway extracts a tenant ID from the Authorization header (JWT sub claim or API key lookup) to identify which token budget to check. Fourth, token budget admission: the gateway checks the tenant's remaining token budget against a distributed counter (Redis or an in-process token bucket synced across gateway replicas) and rejects with HTTP 429 if the budget is exhausted — this check must be fast, under 1ms, or it becomes a bottleneck. Fifth, queue-depth-aware backend selection: the gateway queries each candidate GPU pod's queue depth metric — vLLM exposes `vllm:num_requests_waiting` — and selects the backend with the shortest pending queue, not a round-robin choice. Sixth, request proxy to GPU pod: the gateway opens an upstream connection and forwards the request. Seventh, streaming response interception: the gateway forwards server-sent events (SSE) tokens back to the client while counting the completion tokens in the stream, recording TTFT at the first data event and tokens-per-second throughout. Eighth, budget decrement: after the stream closes or the request completes, the gateway decrements the tenant's token budget by the actual prompt plus completion token count reported in the final usage field.

## Q: What can go wrong with inference routing?

**Answer:**

Four failure modes have high blast radius in production inference systems. First, GPU queue buildup: if the gateway uses round-robin load balancing instead of queue-aware routing, requests pile up unevenly. One GPU pod accumulates a queue of 50 pending requests while another is idle, because each request's processing time is highly variable. The symptom is p95 and p99 TTFT degrading dramatically while p50 looks normal — the tail is the queue, not the GPU. The mitigation is switching to least-request or queue-depth-aware routing and alerting on `vllm:num_requests_waiting` exceeding a threshold per pod. Second, noisy neighbor token exhaustion: without per-tenant token budgets enforced at the gateway, a single tenant running a batch workload (e.g., a nightly document summarization job generating millions of tokens) can saturate all GPU capacity and block interactive workloads. The mitigation is separate token buckets per tenant with different replenishment rates for batch vs. interactive classifications. Third, canary missing quality regressions: a canary rollout shifts 10% of traffic to a new model version, and latency and error rate metrics look identical to the baseline. But the new model produces lower-quality outputs — shorter responses, hallucinations, degraded instruction following. Operational metrics alone miss this. The mitigation is routing shadow traffic to an automated evaluation harness that scores outputs on a held-out benchmark and gates the canary promotion on quality scores. Fourth, KV cache thrashing from random load balancing: many inference backends maintain a KV cache that stores the computed attention states for recently seen prompt prefixes. If requests from the same session or using the same system prompt are scattered across all pods by round-robin, every pod must recompute the KV cache from scratch. The symptom is elevated GPU compute utilization and inflated TTFT despite low queue depth.

## Q: How would you debug high time-to-first-token in production?

**Answer:**

I approach TTFT debugging in three layers: the GPU backend, the gateway queue, and the network path. At the GPU backend layer, I start with `vllm:num_requests_waiting` — if this gauge is above 5-10 on any pod, that pod has a queue and requests are waiting for the scheduler to batch them. The vLLM continuous batching scheduler prioritizes filling the current batch to its `max_num_batched_tokens` limit before issuing prefill for new requests, so a large active batch delays new arrivals. I also check `vllm:gpu_cache_usage_perc`: if the KV cache is above 90% utilized, the scheduler is evicting or swapping existing sequences to disk (if swap is configured), which adds significant latency. High `vllm:num_preemptions_total` confirms evictions are happening. At the gateway queue layer, I look at the gateway's own `upstream_rq_pending_total` stat in Envoy or equivalent. If the upstream cluster's connection pool is exhausted — controlled by `max_requests` in Envoy's circuit breaker — requests queue at the gateway itself before even reaching the GPU pod. I check `upstream_rq_pending_overflow` to confirm. I also check whether the gateway is balancing correctly: `upstream_rq_total` per endpoint should be proportional to capacity, not skewed. At the network layer, I check whether the TTFS (time-to-first-send) from the gateway's perspective equals the TTFT the client sees — if there is a gap, there is a buffering problem in the streaming path (chunked transfer encoding buffering, gzip middleware processing the full body before forwarding). Real TTFT is measured as the time from request send to the first `data:` line in the SSE stream, not the time to full response, and this distinction matters enormously when debugging.

## Q: How would you apply this in a platform engineering context?

**Answer:**

Building a multi-tenant inference platform means solving the same problems as multi-tenant compute, plus the unique constraints of GPU scarcity and non-uniform request cost. The model catalog layer is analogous to a container registry: each model version is an immutable artifact with a version tag, and the platform maintains a catalog CRD that maps model names and versions to serving endpoints. The gateway reads the catalog to resolve `gpt-4-turbo` to the current serving deployment, supporting blue-green model deployments without client-side changes. For per-team token budgets, I implement a two-level quota system: each team has a monthly token allocation set in a quota CRD, and the gateway enforces a per-minute burst rate against a Redis token bucket, backed by a nightly reconciler that resets buckets and reports consumption against the monthly allocation. Teams approaching their limit get a warning header in responses before hard rejection begins. For GPU quota management, model serving deployments use a custom scheduler plugin or node affinity rules that pack GPU pods onto nodes efficiently, combined with horizontal pod autoscaling driven by `vllm:num_requests_waiting` rather than CPU — CPU is not correlated with GPU inference demand. For canary rollouts, I define a promotion pipeline: deploy the new model version at 5% weight, run automated evaluation against a golden dataset for 30 minutes, compare TTFT p99 and tokens-per-second against the baseline, check quality scores, and either promote to 100% or trigger automatic rollback. The rollback mechanism is a weight update to the gateway's routing config, not a deployment rollback, so it completes in seconds. I also maintain shadow traffic routing: every production request is duplicated to a staging model version at zero cost to the client, allowing pre-production validation with real traffic distributions.

## Q: What is the difference between request-rate limiting and token-rate limiting?

**Answer:**

Request-rate limiting counts API calls per unit of time — 100 requests per minute, regardless of what those requests contain. Token-rate limiting counts the input plus output tokens consumed per unit of time. The difference matters enormously for inference workloads because the cost of an inference request is almost entirely determined by token count, not request count. A request with a 100-token prompt that generates a 40-token completion consumes roughly the same GPU time as running a small matrix multiply. A request with a 32,000-token context window and a 4,000-token completion occupies a GPU for several seconds and consumes 100x more KV cache memory. If you rate-limit by request rate, a single user can send one request per minute — well within any reasonable request-per-minute limit — and still consume the entire GPU capacity of a serving cluster for minutes at a time with pathological token counts. This is why OpenAI, Anthropic, and every major inference provider rate-limits by tokens-per-minute (TPM) and tokens-per-day (TPD) rather than by request rate alone. The implementation difference is also significant: request-rate limiting can use a simple counter with a sliding window; token-rate limiting requires knowing the token count before or after the request. For input tokens, the count can be estimated before sending (tokenize the prompt client-side or at the gateway). For output tokens, the count is only known after the generation completes. This means token budget enforcement must either use a pre-flight estimate based on the `max_tokens` parameter in the request, decrement optimistically and reconcile after completion, or combine both approaches — pre-flight check against `max_tokens` cap, post-completion decrement against actual usage. The `max_tokens` pre-flight approach is simpler and prevents runaway generation from exhausting budgets.

## Q: How would you design a canary rollout for a new model version?

**Answer:**

A model canary rollout differs from a service canary in two critical ways: quality regression is not captured by operational metrics alone, and rollback means changing routing weights rather than rolling back a Deployment. The rollout pipeline I use has five phases. First, shadow phase: deploy the new model version and route 100% of production traffic to it as shadow requests — the real responses still come from the current model, but the new model processes every request in parallel and its outputs are logged for evaluation. This runs for at least 24 hours to sample diverse traffic. An automated evaluator scores the shadow outputs against the production outputs on a set of quality metrics — task completion rate on instruction-following benchmarks, response length distribution, refusal rate. Second, canary admission: if shadow quality metrics are within 5% of production baseline and operational metrics (TTFT p99, tokens-per-second, error rate) are within acceptable bounds, the pipeline promotes to a live canary at 5% traffic weight using the gateway's `weighted_clusters` configuration. Third, canary observation window: the canary runs at 5% for a minimum observation period (typically 30-60 minutes for stateless workloads, longer for models with session memory). The system monitors TTFT p99, error rate, and an LLM-as-judge quality score computed on sampled canary outputs against a golden evaluation set. Fourth, staged promotion: on success, weight advances through 5% → 20% → 50% → 100% with automated observation windows at each stage. Fifth, automated rollback: if any metric exceeds its threshold at any stage — TTFT p99 increases by more than 20%, error rate exceeds 1%, quality score drops below threshold — the pipeline immediately sets the canary weight back to 0 and pages the on-call. The rollback takes effect in the time it takes the gateway to reload its routing config, typically under 5 seconds.

## Q: What is KV cache affinity and when does it matter?

**Answer:**

KV cache affinity is a routing strategy that sends requests likely to share prompt prefix context to the same GPU pod, exploiting the inference backend's ability to reuse previously computed key-value attention states. During transformer inference, computing the attention mechanism for each token in the prompt — the prefill phase — is the most compute-intensive step. Modern inference servers like vLLM implement prefix caching: if the first N tokens of a new request's prompt are identical to a recently processed request, the server skips recomputing their KV states and reads them from the cache instead. This can reduce TTFT by 50-80% for requests with long shared prefixes — the common case being a system prompt that is identical across all requests from the same application. Without routing affinity, a gateway using round-robin distributes requests across all GPU pods. Pod A computes and caches the system prompt KV states, but the next request from the same application is sent to pod B, which has a cold cache and must recompute from scratch. With prefix-aware routing, the gateway hashes the first K tokens of the prompt (or uses the session ID for conversational workloads) and routes consistently to the same pod, maximizing cache hits. The trade-off is load imbalance: if all traffic from a high-volume application is pinned to one pod, that pod becomes a hot spot while others are underutilized. The mitigation is consistent hashing with a small number of replicas per hash key — rather than pinning to one pod, the request hashes to one of three candidate pods and the least-loaded of those three is selected (consistent hashing with bounded loads). KV cache affinity matters most for agentic workloads with long system prompts, multi-turn conversations with growing context, and retrieval-augmented generation patterns where the retrieved document chunks are prepended identically across many requests. It matters least for single-turn requests with short or unique prompts, where the cache hit probability is low regardless of routing strategy.
