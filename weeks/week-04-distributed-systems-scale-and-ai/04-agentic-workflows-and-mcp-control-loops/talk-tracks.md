# Talk Tracks

## Q: Explain agentic workflows in one minute.

**Answer:**

An agentic workflow is a control loop driven by a language model rather than hardcoded logic. It maps directly onto the Kubernetes reconciliation pattern: the agent observes current state by calling read-only tools, plans a sequence of actions by reasoning over what it observed, acts by calling write tools, verifies that the intended outcome was achieved, and reconciles any remaining gap before the loop completes. The key insight is that every safety property that makes Kubernetes controllers reliable — idempotent operations, level-triggered observation, optimistic concurrency, retry with backoff, explicit status — applies equally to agent-driven automation. The difference from a controller is that the plan is generated at runtime by an LLM rather than compiled into code, which makes the tool permission boundaries and dry-run validation more important, not less. Safety comes from three things: least-privilege tool scoping enforced server-side in the MCP tool handler (not in the LLM prompt), a dry-run gate before every write that lets the agent verify the diff matches the plan, and a human approval gate for any action above a configurable blast-radius threshold. MCP (Model Context Protocol) is the standardized JSON-RPC interface that connects the LLM to its tools, playing the same role as xDS does for Envoy: it decouples the reasoning layer from the tool implementations and enables consistent audit logging, schema validation, and scope enforcement across all tool calls.

## Q: Walk me through the internals of an agent control loop.

**Answer:**

The loop has five phases, each with a distinct role and safety invariant. In the observe phase, the agent calls only Tier 1 read tools — `kubectl_list`, `metrics_query`, `logs_fetch` — to build a ground-truth snapshot of the environment. This phase is always idempotent, never requires confirmation, and must be level-triggered: the agent re-reads current state at the start of every loop iteration rather than trusting a snapshot from a prior run. In the plan phase, the agent reasons over the observation document and generates a structured JSON plan listing each write operation with its expected before-state, expected after-state, and rationale. The plan is not executed during this phase — it is written to the reconciliation trace so it can be reviewed and validated. In the dry-run phase, the agent calls every planned write tool with `dry_run: true`, submitting each manifest to the Kubernetes API server to run admission webhooks and validation without writing to etcd. If the dry-run result does not match the expected after-state — because a webhook mutated the object or a validation error returned — the agent halts and replans before proceeding. In the act phase, the agent executes the write operations in dependency order — Gateway before HTTPRoute, new routes before deleting old Ingress — writing each action to the audit log before the call is made. Between the plan and act phase, the blast-radius calculator evaluates the scope of the change: if it exceeds the threshold, a `request_human_approval` tool call blocks the loop until a human approves. In the verify phase, the agent calls read tools and HTTP probes to confirm the outcome matches intent — not just that the objects exist, but that traffic is actually flowing. A failed verify triggers automatic rollback execution.

## Q: What can go wrong with agentic automation?

**Answer:**

Four failure modes dominate. First, non-idempotent action repetition: if the agent uses `kubectl create` instead of `kubectl apply`, a re-run after a partial failure creates duplicate resources instead of converging to correct state. The mitigation is to enforce idempotent semantics at the tool layer — `kubectl apply` for all write operations, and an observe phase that detects prior partial runs. Second, drift between plan and act: the agent observed state at T0 and planned at T1, but between T1 and T2 (when it applies), another actor mutated the target objects. The apply succeeds but produces a state the agent never validated. The mitigation is to use optimistic concurrency — embed the `resourceVersion` from the plan phase in every write, so the API server rejects writes with a 409 Conflict if the object changed since observation. Third, permission escalation via tool arguments: if scope constraints live only in the LLM prompt ("only modify the app namespace"), a prompt injection attack or a model hallucination can generate a tool call targeting `kube-system`. The mitigation is server-side enforcement — the MCP tool handler must reject out-of-scope namespace arguments regardless of the LLM's instructions. Fourth, infinite retry loop: the agent fails verification, re-observes, re-plans the same apply, and loops indefinitely, generating LLM inference cost, API server load, and an unreadable audit trail. The mitigation is a maximum retry budget per loop, a `wait_and_verify` tool for convergence delays like DNS propagation, and a circuit breaker that halts the loop if the same plan has executed N times without progress.

## Q: How would you debug an agent that is making incorrect changes?

**Answer:**

I start with the reconciliation trace, which is the structured log the agent writes before and after every tool call. The trace shows the observation document the agent built, the plan it generated, the dry-run results, and the actual tool call arguments and responses. If the trace shows the plan was correct but the apply produced a different result, the issue is drift between plan and apply — I check the API server audit log for concurrent mutations in the target namespace during the same time window using `kubectl logs -n kube-system deploy/kube-apiserver | grep '"namespace":"app"' | grep '"verb":"update"'`. If the trace shows the plan itself was wrong — the agent planned to create a resource with incorrect parameters — I check whether the observation document was complete: did the observe phase actually read all the relevant state, or did it miss a field that caused the plan to be based on incomplete information? For scope-related errors I check the MCP server logs for scope enforcement hits: `kubectl logs -n agent-system deploy/mcp-server | jq 'select(.scope_check == "rejected")'`. For repeated incorrect changes with no convergence, I check the loop iteration count in the trace and look for the pattern where the same plan generates the same apply and the same failed verification on every iteration, which points to a missing `wait_and_verify` tool or a verification check that is testing the wrong condition. For Gateway API routing issues specifically, I check `kubectl get gateway -n app -o jsonpath='{.status.conditions}'` and `kubectl describe httproute` to see the Gateway controller's programming status, and then run `curl -H "Host: app.example.com" http://<gateway-ip>/` to verify end-to-end routing independently of the agent's own probes.

## Q: How would you apply this in platform engineering?

**Answer:**

The clearest application is automating routine but error-prone migration work: translating Ingress resources to Gateway API HTTPRoutes, upgrading Helm chart versions across a fleet of clusters, rotating certificates before expiry, or triaging and remediating common alert patterns. These tasks share a structure that maps perfectly to the agent loop: they require multi-step observation, diff-based planning, and verification that goes beyond "did the kubectl apply succeed." For Ingress-to-Gateway migration specifically, an agent can discover all Ingress objects cluster-wide, translate each one to the equivalent HTTPRoute and Gateway configuration including TLS passthrough and path rewrite annotations, run dry-run validation against the live cluster to catch schema incompatibilities, apply to staging, run synthetic HTTP probes to verify routing, and open a PR to remove the original Ingress resources after a 48-hour soak — all without human involvement except at the blast-radius gate for production apply. The safety architecture maps onto platform engineering constraints: the agent's MCP server runs under a ServiceAccount with namespace-scoped RBAC, blast-radius thresholds are set per environment (e.g., staging allows up to 100% traffic rerouting without gate, production requires approval above 5%), and the audit trail feeds directly into the change management system. The economic argument is also real: for a fleet of 50 clusters each with 200 Ingress objects, manual migration is weeks of error-prone work; an agent with the right safety envelope completes it in hours with a full audit trail and automatic rollback on any verification failure.

## Q: What is MCP and how does it relate to Kubernetes patterns?

**Answer:**

MCP, the Model Context Protocol, is an open standard for LLM-to-tool communication. It defines a JSON-RPC protocol where an MCP server exposes a `tools/list` endpoint that returns structured tool descriptors — each with a name, description, and JSON Schema for its input parameters — and a `tools/call` endpoint that executes a named tool with validated arguments and returns a structured result. The LLM runtime acts as the MCP client: it receives the tool list at session initialization, generates tool call requests as part of its reasoning, and receives tool results as continuation messages in its context. The architectural parallel to Kubernetes is precise. MCP plays the same role as xDS plays for Envoy: just as xDS decouples Envoy's data plane from its specific control plane by standardizing the resource discovery protocol (allowing Istio, Consul, or any xDS-compatible server to control the same Envoy), MCP decouples the LLM's reasoning from specific tool implementations, allowing the same agent prompt to work with different MCP server backends — one wrapping `kubectl`, another wrapping Terraform, another wrapping a metrics API. The security model parallel is also precise: xDS authentication uses mTLS and RBAC at the control plane level to limit what Envoy configurations a given server can push; MCP security uses server-side scope enforcement and the backing ServiceAccount's Kubernetes RBAC to limit what operations a given tool call can perform. Both solve the same problem: the client (Envoy or LLM) must not be trusted to self-limit based on instructions alone — the server must enforce boundaries independently.

## Q: How do you ensure safety in agentic automation?

**Answer:**

Safety in agentic automation has four independent layers that must all be present — defense-in-depth, not alternatives. The first layer is tool-level RBAC: the MCP server's Kubernetes client runs under a ServiceAccount with exactly the permissions needed for the task, scoped to specific namespaces and resource types. A migration agent needs `get` and `list` on Ingress, `create` and `update` on HTTPRoute and Gateway, and nothing else. If the ServiceAccount cannot physically delete cluster-scoped resources, no LLM instruction or prompt injection can cause the agent to delete them. The second layer is server-side scope enforcement in the MCP tool handler: every write tool validates that the target namespace and resource type are in the tool's configured allowlist before passing the call to the Kubernetes client. This is enforced in code, not in the LLM prompt. The third layer is a mandatory dry-run gate before every write: the agent calls `kubectl apply --dry-run=server` for every manifest, submitting it to the full admission and validation stack without writing to etcd. The dry-run result is compared against the expected after-state from the plan; a mismatch causes replanning before any write is executed. The fourth layer is a blast-radius-gated human approval workflow for destructive or high-impact actions: a `request_human_approval` tool blocks the loop and presents the plan and blast-radius estimate to an operator, who must explicitly approve before the loop continues. The gate defaults to rejection on timeout. Critically, none of these layers replaces the others: RBAC without dry-run allows permission-compliant but incorrect changes; dry-run without RBAC still allows a scope-escalation bug to reach the API server; and approval gates without RBAC enforce process but not access control.

## Q: Walk me through an agent-driven migration from Ingress to Gateway API.

**Answer:**

The migration agent runs a six-phase workflow. In the observe phase, the agent calls `kubectl_list` for all Ingress resources in the target namespace, reads each Ingress spec including host, path rules, backend service references, and TLS configuration, then calls `kubectl_list` for existing HTTPRoute and Gateway objects to detect prior partial runs. It also reads the `httproutes.gateway.networking.k8s.io` CRD to confirm Gateway API v1 is installed and served. In the plan phase, the agent generates a structured JSON plan: create one Gateway resource as the parent for all routes, then create one HTTPRoute per Ingress preserving the host, path match rules (translated from Ingress pathType `Prefix` to HTTPRoute `PathPrefix`), backend service references, and TLS parent references. Path rewrites and annotation-based configurations from ingress-nginx (e.g., `nginx.ingress.kubernetes.io/rewrite-target`) are translated to HTTPRoute filter rules. The rollback plan is generated in parallel: delete all created HTTPRoutes and Gateway, verify original Ingresses are intact. In the dry-run phase, the agent submits each manifest with `kubectl apply --dry-run=server` and checks for admission errors or unexpected mutations. In the act phase, the agent applies the Gateway first, waits for its `Programmed: True` status condition, then applies each HTTPRoute in order, logging each write to the audit trail before execution. After all writes complete, the verify phase runs HTTP probes against each hostname and path using `curl -H "Host: app.example.com" http://<gateway-ip>/` and checks that all probes return the expected status codes. On success, the agent opens a pull request to remove the original Ingress objects after a 48-hour observation window. If any probe fails, the agent executes the rollback plan, deletes the HTTPRoutes and Gateway, and pages the on-call engineer with the full reconciliation trace attached.
