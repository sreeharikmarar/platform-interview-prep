# Talk Tracks

## Q: Present your platform design in one minute.

**Answer:**

The platform separates developer intent from infrastructure realization using a compiler-style architecture. Developers declare what they want via CRDs — AppService for workloads, InferenceService for AI models, TenantNamespace for isolation boundaries. Platform controllers on a hub cluster compile these declarations into concrete infrastructure: Kubernetes Deployments, Envoy xDS routes, NetworkPolicies, RBAC bindings, and Prometheus monitoring rules. The hub pushes generated resources to spoke clusters via ArgoCD ApplicationSets, with each spoke running workloads in its own failure domain. Multi-tenancy is enforced through namespace isolation with defense-in-depth: Kubernetes RBAC scopes API access, NetworkPolicy enforces L3/L4 boundaries, Istio AuthorizationPolicy enforces L7 identity-based access using SPIFFE SVIDs, and ResourceQuota caps resource consumption. Lifecycle management follows progressive delivery — changes roll out through a canary spoke cluster first, are validated against SLO metrics, then propagate to production spokes in waves. Observability is a platform primitive: tenants get Grafana dashboards, PrometheusRules, and distributed tracing automatically when they onboard. AI workload support is built in through an inference gateway — a dedicated Envoy deployment with token-rate limiting, model-version routing, and GPU quota enforcement. The key architectural decision is treating infrastructure config as a compilation target rather than a manual artifact.

## Q: Walk me through the architecture end-to-end.

**Answer:**

A developer pushes an AppService manifest to Git. ArgoCD detects the change and syncs it to the hub cluster. The AppService controller on the hub reads the CRD, resolves dependencies like the team's TenantNamespace for RBAC context, and generates child resources: a Deployment with resource limits and security context, a Service, a VirtualService for Envoy routing, a NetworkPolicy allowing ingress only from the team's other services and the ingress gateway, a ServiceMonitor for Prometheus scraping, and a GrafanaDashboard. These child resources are written to the hub etcd with owner references pointing back to the AppService. ArgoCD ApplicationSets, templated from the Cluster Registry's SpokeCluster CRs, detect the new child resources and sync them to the target spoke cluster based on placement labels in the AppService spec. On the spoke, the Envoy sidecar injector adds a proxy to the Deployment pods, Istiod on the hub pushes xDS configuration to the sidecar via gRPC stream, and the pod starts receiving traffic. The full propagation path is Git to hub in about 30 seconds via ArgoCD polling, hub reconciliation in under 5 seconds, and ArgoCD sync to spoke in another 30 seconds — so roughly one minute from commit to running pod. Each component in this chain is independently recoverable: if the hub controller crashes, it restarts and re-reconciles from the CRD state in etcd.

## Q: What can go wrong?

**Answer:**

The highest-blast-radius failure is a bad platform controller release that generates invalid child resources. If the controller emits VirtualServices with empty hosts, Istiod rejects them at the xDS layer, but the delete-and-recreate pattern means the old valid routes are removed before the new invalid ones fail validation — leaving all Envoy proxies with empty route tables and all tenant traffic dropped. Mitigation requires the controller to patch in-place rather than delete-and-recreate, and to validate generated objects via dry-run before writing. Second, multi-cluster drift occurs when a spoke's state diverges from hub intent — a Kyverno policy version mismatch on one spoke injects a stale environment variable, causing CrashLoopBackOff only on that spoke while the hub status shows Synced. Detection requires the platform controller to actively probe spoke-level pod health rather than trusting ArgoCD sync status alone. Third, CRD proliferation creates operational debt — one CRD per feature rather than composable primitives leads to hundreds of types, controller memory bloat, and etcd growth. Fourth, GPU capacity exhaustion after spot instance reclaims leaves InferenceService pods in Pending while the quota system still shows headroom because it was calculated against the original node count rather than real-time allocatable capacity.

## Q: How would you debug a tenant whose service is unreachable?

**Answer:**

I trace through the data path layer by layer. First DNS: does the hostname resolve? I check if the VirtualService exists and has the correct host, and whether Istiod has pushed the route to the Envoy sidecar using `istioctl proxy-config route <pod>`. If the route exists, I check the cluster configuration — `istioctl proxy-config cluster <pod>` shows whether endpoints are populated. If endpoints are missing, I check the Service and EndpointSlice in the spoke namespace. If endpoints exist but traffic fails, I check NetworkPolicy — is there a policy that denies ingress from the gateway or from the calling service's namespace? I verify with `kubectl get networkpolicy -n <ns>` and check if the pod labels match the policy's podSelector. If network is open, I check the Envoy access log for the specific request — the response flags field tells me if it was a `NR` (no route), `UH` (upstream unhealthy), `UO` (upstream overflow from circuit breaker), or `DC` (downstream connection termination). For mTLS failures, I check `istioctl authn tls-check` between the source and destination to verify both sides agree on the TLS mode. Throughout this process I'm also checking the AppService status conditions on the hub — if `SpokeHealthy` is False, the platform controller has already detected the issue and the condition message usually points to the root cause.

## Q: How would you apply this in your current role?

**Answer:**

In my current platform engineering context, the direct applications are threefold. First, the intent-vs-realization pattern applies to our gateway migration work — instead of teams manually configuring Ingress resources with nginx-specific annotations, we can define a platform CRD that captures routing intent (hostname, paths, backends, TLS mode) and have a controller generate the correct Gateway API HTTPRoutes, eliminating the annotation drift problem. Second, the hub-spoke model maps to our multi-cluster topology — a hub cluster running platform controllers that generate and push standardized configurations to regional spoke clusters ensures consistency without requiring each team to understand cluster-specific details. Third, the inference gateway integration is directly relevant as we onboard AI workloads — wrapping GPU scheduling, token budget enforcement, and model routing into a single InferenceService CRD makes the platform self-service for ML teams instead of requiring platform engineer involvement for each model deployment. The progressive rollout pattern also applies to how we ship platform changes: canary to a staging spoke, validate metrics, then wave-based rollout to production spokes.

## Q: How do you handle multi-tenancy isolation?

**Answer:**

Multi-tenancy is enforced through four independent layers, any one of which would limit blast radius even if the others failed. The first layer is namespace isolation with RBAC — each team gets a dedicated namespace via the TenantNamespace CRD, and the platform controller generates Role and RoleBinding resources that grant the team's OIDC group edit access only within their namespace. ClusterRole creation and admission webhook modification are explicitly excluded. The second layer is network isolation — a default-deny NetworkPolicy is applied to every tenant namespace, and the platform controller generates allow rules only for declared service dependencies in the AppService spec. This means a compromised pod in team-a's namespace cannot reach team-b's pods at the network level. The third layer is identity-based authorization — Istio AuthorizationPolicy checks the SPIFFE SVID of the calling workload, not just its IP address, so even if a pod spoofs its source IP to bypass NetworkPolicy, the mTLS identity check catches it. The fourth layer is resource isolation — ResourceQuota caps CPU, memory, and GPU per namespace, and LimitRange sets per-pod defaults, preventing a single tenant from starving others. For AI workloads, InferenceQuota tracks GPU allocation separately. The platform team monitors cross-tenant network flow logs and alerts on any traffic that doesn't match a declared dependency.

## Q: How do you roll out platform changes safely?

**Answer:**

Platform changes follow a strict five-step progressive rollout. Step one is hub dry-run: the new controller image deploys to a canary namespace with dry-run mode enabled — it reads all CRD objects, computes the desired child resources, and logs diffs to stdout without writing anything. This catches parsing errors and nil pointer panics against real production data. Step two is hub canary: the new image handles 5% of reconcile events (selected by deterministic hash of object name modulo 20) alongside the current image. Error rate in `platform_reconcile_errors_total` with the version label is monitored for 30 minutes. Step three is hub full rollout: the old image scales down and the new image handles all events. Child resource generation is complete but spoke clusters are not yet updated. Step four is spoke wave one: ArgoCD syncs the updated manifests to canary-tier spoke clusters (5% of fleet). SLO dashboards for those spokes are reviewed for 15 minutes. Step five is spoke full rollout: remaining spokes are updated in waves of 10%, with a minimum 15-minute hold between waves. The ApplicationSet sync policy pauses rollout if `platform_reconcile_errors_total` increases by more than 5% on any spoke during its hold window. CRD schema changes follow the same pattern with an additional conversion webhook validation step. The entire rollout can be rolled back at any step by reverting the Git commit, which ArgoCD syncs automatically.

## Q: How does the inference gateway fit into the platform?

**Answer:**

The inference gateway is a platform service declared via the InferenceService CRD. When a team creates an InferenceService specifying model name, version, GPU requirements, and token budget, the platform controller generates five resources. First, a GPU-scheduled Deployment with an init container that pulls the model artifact from object storage, a vLLM or TGI serving container with the appropriate GPU limits, and a readiness probe that checks model load completion. Second, an xDS route update pushed to the inference gateway's Envoy deployment — a dedicated Envoy instance separate from the tenant ingress gateway, configured with longer idle timeouts and HTTP/2 streaming support for inference responses. Third, an ext_authz filter configuration that routes each request through the platform's token budget enforcement service — a gRPC service backed by Redis that tracks per-tenant token consumption and returns 429 when the budget is exhausted. Fourth, a ResourceQuota update that decrements the team's GPU allocation. Fifth, a GrafanaDashboard with inference-specific metrics: TTFT histogram, tokens per second, queue depth, and remaining token budget with an alert at 10%. The inference gateway is the single entry point for all AI traffic, enabling centralized token budget enforcement, cost attribution logging, and per-model canary routing — shifting 10% of traffic to a new model version while monitoring quality metrics alongside the operational ones.

## Q: How would you evolve this platform over 3 years?

**Answer:**

Year one focuses on the foundation: namespace-per-team isolation, a single hub cluster with 3-5 spoke clusters, AppService and TenantNamespace CRDs, basic progressive delivery via ArgoCD, and Prometheus-based observability with auto-generated dashboards. The gateway runs nginx-ingress with plans for Gateway API migration. The goal is to get developer self-service working end-to-end so teams can deploy without filing tickets. Year two adds sophistication: Gateway API replaces Ingress, Istio service mesh enables mTLS and L7 policy, the inference gateway launches for AI workloads with token budgets, the platform agent automates routine migrations and capacity management, and multi-cluster expands to 10-15 spokes across regions. CRD versioning and conversion webhooks become essential as the schema evolves. Year three focuses on intelligence and scale: the agent handles incident correlation and automated remediation, predictive capacity planning uses historical usage data to right-size GPU pools before demand spikes, the platform supports 50+ spoke clusters with tiered SLOs, and the observability pipeline evolves from metrics-first to traces-first with OpenTelemetry. The key principle throughout is to resist the temptation to build everything in year one — start with the simplest version that unblocks developers, then add layers of sophistication as the platform proves its value and the team builds operational confidence.
