# Talk Tracks

## Q: Explain cgroups and namespaces in one minute.

**Answer:**

Containers are built from two independent Linux kernel primitives. Namespaces control visibility: what resources a process can see. The eight namespace types isolate PID numbers, network interfaces, mount tables, hostnames, IPC objects, user IDs, cgroup roots, and system clocks. A container process is placed in a fresh set of namespaces at creation via `clone(2)` with flags like `CLONE_NEWPID | CLONE_NEWNET | CLONE_NEWNS`, so it sees only its own processes, its own network stack, and its own filesystem view. cgroups control consumption: how much CPU, memory, I/O, and device access a process tree is allowed to use. The kubelet writes limits from the Pod spec into pseudo-files like `memory.max` and `cpu.max` under `/sys/fs/cgroup/kubepods.slice/` per container. When the container exceeds its memory limit, the kernel OOM kills a process within that cgroup. When it exceeds the CFS CPU quota, the kernel throttles its threads until the next scheduling period. Neither primitive knows about the other — isolation failures in one don't affect the other — which is why debugging requires identifying first whether the symptom is a visibility problem (namespace) or a resource problem (cgroup).

## Q: A container is using 40% CPU but is throttled — how?

**Answer:**

This is the CFS bandwidth controller period problem, and it is one of the most common misconfigured resource constraints in production Kubernetes. The Linux CFS scheduler does not measure CPU utilization as a rolling average — it enforces a hard quota within each period. When you set `cpu: 500m` in a Pod spec, the kubelet writes `cpu.max = 50000 100000` to the container's cgroup, meaning the container can consume 50ms of CPU time per 100ms period. If the application is bursty — for example, a Go HTTP server that spends 20ms handling a request (using all its quota in one goroutine burst), then idles — it will exhaust the 50ms quota in those 20ms and be throttled for the remaining 80ms of that 100ms period, even though its average over a minute is only 20%. The throttling is visible in `container_cpu_throttled_periods_total` and in `cpu.stat`'s `throttled_usec` field in the cgroup. Latency-sensitive applications show this as periodic p99 spikes at exactly 100ms intervals, often confused with garbage collection pauses. The mitigation paths are: increase the CPU limit to cover peak burst rather than average usage, increase the CFS period via `--cpu-cfs-period` on the kubelet (longer period = larger burst window), or for latency-critical pods, disable CFS quota enforcement entirely with `--cpu-cfs-quota=false` and rely on `requests` for scheduling placement while allowing the pod to burst to available node CPU.

## Q: Explain cgroups v1 vs v2 — when does it matter?

**Answer:**

cgroups v1 organizes resource controllers as completely independent hierarchies. The memory controller has its own tree rooted at `/sys/fs/cgroup/memory/`, the CPU controller has a separate tree at `/sys/fs/cgroup/cpu/`, and a process can be in different groups in each hierarchy simultaneously. This independence creates accounting problems: you cannot atomically measure a process's combined resource usage across controllers, because the process's identity is different in each hierarchy. It also makes Docker-in-Docker broken for resource limits — an outer container cannot accurately impose memory limits on processes in an inner container because the inner container's cgroup hierarchy is invisible to the outer's memory controller accounting. cgroups v2 unifies everything under a single tree at `/sys/fs/cgroup/`. A process is in exactly one group for all controllers simultaneously. This enables accurate per-container resource attribution regardless of nesting, makes the `memory.events` file report comprehensive OOM statistics including kills of nested processes, and enables the `memory.oom.group` feature which kills all processes in a cgroup together rather than selecting victims individually. For Kubernetes, it matters starting with 1.25, which promoted cgroupsv2 support to GA and set it as the default for kubeadm clusters on nodes running unified cgroup hierarchy. It also matters for any workload using Docker-in-Docker (CI/CD systems, Tekton) because v2 correctly propagates resource limits through nested container runtimes. The practical check: `cat /sys/fs/cgroup/cgroup.controllers` — if this file exists, the node is running cgroupsv2. On v1, that path doesn't exist and you'd check `/sys/fs/cgroup/memory/` for individual controller mounts.

## Q: Trace resources.limits.memory: 256Mi from Pod spec to OOMKilled.

**Answer:**

The path from YAML to OOM kill traverses five layers. First, `kubectl apply` sends the Pod spec to the API server, which validates `256Mi` (268435456 bytes) and stores it in etcd. Second, the scheduler assigns the Pod to a node and kubelet picks up the assignment via its Pod watch. Third, kubelet calls the container runtime (containerd) via CRI to create the container. It passes the memory limit as `linux.resources.memory_limit_in_bytes = 268435456` in the CRI `LinuxContainerResources` struct. Fourth, containerd's shim calls the runc OCI runtime, which writes `268435456` to the container's cgroup: on cgroupsv2 it writes to `memory.max`; on cgroupsv1 it writes to `memory.limit_in_bytes`. Fifth, when a process in the container calls `malloc()` which calls `mmap(2)`, the kernel checks whether `memory.current + requested_bytes > memory.max`. If yes, the kernel first attempts direct reclaim: evicting page cache, writing dirty pages, swapping if swap is enabled. If reclaim cannot free enough bytes, the OOM killer runs within the cgroup scope. It selects the victim by highest `oom_score = RSS_fraction * 1000 + oom_score_adj`. Kubernetes sets `oom_score_adj` based on QoS: Guaranteed pods get `-997` (rarely killed), Burstable pods get a score between 2 and 999, BestEffort get `1000`. The selected process receives SIGKILL. The container runtime detects exit code 137 (killed by signal 9). The kubelet sets `containerStatuses[].state.terminated.exitCode = 137` and `reason = OOMKilled`. `kubectl describe pod` shows `OOMKilled`, and the pod enters `CrashLoopBackOff` if `restartPolicy` permits restarts.

## Q: What does running as UID 0 inside a container but non-root on the host mean, and is it safe?

**Answer:**

When a container runs with a user namespace (`CLONE_NEWUSER`), UID 0 inside the container maps to an unprivileged UID on the host via a UID mapping written to `/proc/<pid>/uid_map`. A mapping of `0 100000 65536` means container UID 0 = host UID 100000, container UID 1 = host UID 100001, and so on. Inside the user namespace, the process has full root capabilities — it can `chown`, `chmod`, create device files, call `mount(2)` — but these capabilities are scoped to the user namespace. When the process tries to use a capability that requires interacting with host resources (writing to a host-owned file, calling `sys_admin` outside the namespace), the kernel checks whether the host UID (100000) has permission, not the container UID. This is the rootless container security model used by podman and rootless containerd. Without a user namespace — the default in most Kubernetes clusters — UID 0 inside the container is the same UID 0 as on the host. The process is root. If it escapes the container (via a kernel exploit, a misconfigured host path mount, or a privileged capability like `SYS_ADMIN`), it has root on the node. This is why `runAsNonRoot: true` and `runAsUser: 1000` in the Pod security context are defense-in-depth measures that matter even though they don't prevent all escapes: a process running as UID 1000 that escapes still only has UID 1000 on the host. The practical safety threshold: always set `runAsNonRoot: true`, avoid `privileged: true` and `CAP_SYS_ADMIN` in production workloads, and push toward user namespace pods (`hostUsers: false`) on Kubernetes 1.30+ nodes for the full rootless-on-host guarantee.

## Q: Walk me through nsenter — how does kubectl exec actually work?

**Answer:**

`kubectl exec` works by joining the namespaces of the target container's init process without creating a new container or any new namespaces. The full path: kubectl sends an Exec subresource request to the API server, which proxies it to the kubelet via the SPDY or WebSocket upgrade protocol. The kubelet identifies the target container's PID — specifically the PID of the container's shim process or the first process in the container's PID namespace — using containerd's CRI `ExecSync` or `Exec` API. containerd's shim then calls the runc `exec` command, which under the hood calls `nsenter(1)` with the container's namespace file descriptors. Concretely: for each namespace type to share (PID, Mount, Network, UTS, IPC), runc opens `/proc/<container-pid>/ns/<type>` to get a file descriptor, then calls `setns(fd, nstype)` to switch the current goroutine's namespace context to the container's namespace. After all `setns()` calls complete, runc calls `fork()` and `exec()` to start the requested command (e.g., `/bin/bash`). The resulting process is now a member of the container's namespaces: it sees the container's PID tree, has access to the container's network stack, and sees the container's filesystem overlay. The process appears in the container's PID namespace as a new PID and also appears in the host's PID namespace (under the container's parent PID) because PID namespaces are hierarchical — child namespaces are always visible to parent namespaces. This is why `ps aux` on the host shows all container processes, but `ps aux` inside the container only shows processes in its own namespace.

## Q: What can go wrong with namespaces and cgroups in production?

**Answer:**

Four failure modes stand out from real production incidents. First, CPU throttling at sub-limit utilization: containers set with tight `cpu` limits relative to their burst characteristics will be periodically throttled even when average utilization is low, causing latency spikes in request-handling workloads. This is invisible in standard CPU dashboards which show averages. Second, OOM killing the wrong process: when a Pod has mixed QoS containers (app as Guaranteed, sidecar as Burstable), the higher-scored Burstable sidecar is killed by the OOM killer even when the Guaranteed app caused the memory pressure — because the OOM killer selects by `oom_score`, not by which container triggered the limit. Third, overlay2 inode exhaustion: containers that create large numbers of small files (build artifacts, node_modules) exhaust the host filesystem's inode table even when disk space is ample, causing ENOSPC errors and eventually making the node unable to schedule new pods. Fourth, namespace leak after container crash: if a container's PID 1 exits without proper cleanup, other processes started by `kubectl exec` may hold references to the container's namespaces via open file descriptors in `/proc/<exec-pid>/ns/`, preventing full namespace cleanup and leaking kernel resources. This manifests as growing inotify watch counts or stale entries under `/run/containerd/io.containerd.runtime.v2.task/`. The detection signal is `ls /proc/*/ns/net | sort | uniq -c | sort -rn` — healthy containers should have one entry per network namespace; leaked namespaces show orphaned entries.

## Q: How would you debug a container that is OOMKilled immediately on start?

**Answer:**

Immediate OOMKill on start before the application has served a single request means either the memory limit is too low for the process's startup footprint, or the application is loading a large dataset or allocating a large structure during initialization. I start by checking the memory limit versus the container's typical RSS at startup using `container_memory_working_set_bytes` in Prometheus, filtering by pod phase Pending-to-Running transition. If the container has never successfully started, I look at the `OOMKilled` event via `kubectl describe pod` and the exit code (137 confirms signal 9). To get the baseline footprint, I temporarily raise the limit to something unrestricted (e.g., 4Gi) and monitor `memory.current` in the cgroup during startup: `watch -n0.5 cat /sys/fs/cgroup/$(crictl inspect <id> | jq -r '.info.runtimeSpec.linux.cgroupsPath')/memory.current`. This shows the actual peak allocation during startup. I then set the limit to that peak plus a 50% safety margin. If the startup footprint itself is problematic — a JVM loading 2GB of class files, a Python process loading a 1GB model — the solution is to reduce the startup footprint: use JVM `-Xms` to set a lower initial heap, lazy-load models, or break initialization into smaller chunks. I also check whether the OOM is coming from the container process itself or from the page cache: if `memory.stat`'s `file` field is high relative to `anon`, the container is allocating page cache (reading large files), and the fix may be to increase `--memory-swap-limit` or use a `tmpfs` mount for large working sets rather than reading from overlayfs.

## Q: Explain overlayfs copy-on-write and when it causes performance problems.

**Answer:**

overlayfs implements copy-on-write at file granularity, not block granularity, which is a critical distinction from block-device based CoW systems like BTRFS or ZFS. When a process inside a container opens a file for writing that exists in an image layer (lower directory) but not yet in the container's upper layer, the overlay filesystem driver intercepts the write and triggers `copy_up()`: it copies the entire file from the lower layer to the upper layer before allowing the write to proceed. This copy is synchronous and must complete before the write returns to the application. The performance implication is severe for large files: writing one byte to a 200 MB binary stored in a lower image layer triggers a 200 MB read + 200 MB write sequence on the host filesystem, plus an fsync of the temp file before the atomic rename. In practice, this means that any application that modifies large binaries at runtime — JVM class reloading, Python writing .pyc files to a directory containing large modules, any build tool running inside a container — will experience unexpected write latency on the first write to each file. This is why the canonical advice is "never install packages inside a running container" — `apt install` inside a container causes copy-up of potentially hundreds of files. The right architecture bakes all modifications into image layers (so they land in lower/ as read-only and never trigger copy-up), and only writes application-specific runtime state (logs, temp files, caches) to volumes or ephemeral storage that goes directly to the upper layer without any image layer counterpart. Directory operations are cheaper — overlayfs merges directory entries in memory without copying the whole directory. The most efficient containers are those that write only to paths that don't exist in any lower layer.
