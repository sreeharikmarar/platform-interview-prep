# Talk Tracks

## Q: Explain Kubernetes cloud networking and IP planning in one minute.

**Answer:**

Kubernetes cloud networking on GKE involves three non-overlapping CIDR blocks that must be sized correctly at cluster creation time because they cannot be changed later without cluster recreation or the Multi-CIDR workaround introduced in GKE 1.26. The primary subnet range gives each node its NIC IP. The pod secondary range is allocated in per-node slices — when max-pods-per-node is 110, each node gets a `/24` block of 256 IPs from this range. The service secondary range is where kube-apiserver draws ClusterIP addresses. In VPC-native mode, GCP's Alias IP feature programs these pod CIDR slices directly into the hypervisor fabric as alias IP ranges on each node NIC, which makes pod IPs natively routable within the VPC without any overlay or tunneling. Packets from pods going to the internet hit an iptables MASQUERADE rule on the node that rewrites the pod source IP to the node IP, and then optionally a Cloud NAT gateway that rewrites the node IP to a public IP. The single most common operational failure in GKE is IP exhaustion: a cluster grows over 18 months, the pod secondary range fills up, new nodes cannot join because the nodeIpamController cannot allocate a `/24` block for them, and the cluster cannot scale out during a traffic event. The correct approach is to size the pod secondary range at 2x the maximum node count times 256, verify the math at design time, and monitor secondary range utilization with alerting.

## Q: Design subnetting for a 500-node, 50,000-pod GKE cluster. Show the CIDR math.

**Answer:**

Starting with nodes: 500 nodes in steady state, but plan 2x headroom for autoscaling peaks and rolling upgrades where old and new node pools coexist simultaneously, so design for 1,000 nodes. The primary subnet range holds node IPs — 1,000 nodes need 1,000 IPs, so a `/22` (1,024 IPs, `10.0.0.0/22`) is sufficient; I use a `/20` (4,096 IPs) for room because primary ranges are cheap and nodes are not the constraint. For the pod secondary range: at max-pods-per-node=110, each node gets a `/24` block of 256 IPs. For 1,000 nodes that is 1,000 `/24` blocks = 256,000 IPs total. That requires a `/14` range (262,144 IPs, `10.100.0.0/14`). A `/16` (65,536 IPs / 256 = 256 nodes) would be too small even for the base cluster. The check: 262,144 / 256 = 1,024 nodes maximum — 1,000 nodes fit with 24 blocks to spare. Note that 50,000 pods all fitting into 500 nodes at 100 pods each is fine because each node has 256 IPs allocated even though only 110 are usable for pods. For the service secondary range: 4,096 ClusterIPs is plenty for even large clusters — use `/20` (`10.200.0.0/20`, 4,096 addresses). Final design: primary `10.0.0.0/20`, pod secondary `10.100.0.0/14`, service secondary `10.200.0.0/20`. Verify no overlap between the three ranges and with any peered VPCs. A common mistake is using `/16` for pods assuming 65,000 IPs is "huge" — 65,536 / 256 = 256 nodes, which a 500-node cluster immediately exceeds under load.

## Q: A pod can reach other pods and services but cannot connect to the internet. A pod on a neighboring node on the same subnet can. What is different between them?

**Answer:**

The fact that a pod on a neighboring node can reach the internet immediately tells me this is not a VPC firewall or Cloud NAT availability problem, because both nodes share the same VPC route and the same NAT gateway. The difference must be local to the node. The most likely cause is that the iptables MASQUERADE rule is missing or misconfigured on the affected node. On each node, the ip-masq-agent DaemonSet (or the CNI plugin's own masquerade setup) writes iptables rules in the `POSTROUTING` chain of the `nat` table that rewrite the pod's source IP to the node IP before egress. Without this rule, packets leave the node with source IP `10.100.5.10` (a pod IP). The VPC knows to route return traffic for pod IPs to the correct node, but Cloud NAT, which performs SNAT only on traffic whose source is a node's primary NIC IP, does not recognize the pod IP as something it should translate. The packet arrives at the internet with source `10.100.5.10` (a private RFC1918 address), which is either dropped by the internet gateway or replied to an address that no public router knows how to reach, so no response arrives. The second possibility is Cloud NAT port exhaustion specifically on that node — the NAT gateway has run out of ephemeral source ports for that node's IP (default minimum is 64 ports), which would cause new connection attempts to fail with `EADDRNOTAVAIL`. I would verify by running `iptables -t nat -L POSTROUTING -n` on the affected node and `iptables -t nat -L POSTROUTING -n` on the healthy neighbor, comparing the MASQUERADE rules, and also checking Cloud NAT's port usage metric for both node IPs.

## Q: A customer wants to migrate from a routes-based GKE cluster to VPC-native. What breaks and what is the process?

**Answer:**

Routes-based and VPC-native GKE clusters are architecturally different at the network layer and cannot be migrated in-place — the pod CIDR allocation mechanism, the node NIC configuration, and the cloud routing model are all different. In routes-based mode, GKE installs one static VPC route per node (`node-pod-cidr → node-vm-ip`), which is why the hard limit is 250 nodes (GCP caps static routes per VPC network). There are no secondary ranges, no alias IPs, and pod IPs are not natively in the VPC routing table beyond those static routes. In VPC-native mode, pod IPs are alias IP ranges on the node NIC programmed directly into the hypervisor fabric. The migration itself requires creating a new VPC-native cluster, which means: designing and allocating the secondary CIDR ranges (pod and service), verifying they are non-overlapping with all existing subnets including the routes-based cluster's ranges while the two clusters coexist during migration, and then migrating workloads cluster-by-cluster using blue/green or traffic shifting. What breaks during the migration window: any hardcoded assumptions about pod IP ranges (firewall rules scoped to the old pod CIDR, security group rules in external systems referencing the cluster's IP range, metrics or log correlation based on IP prefix), any VPC peering configurations that included the old routes-based static routes, and any network policies using IP-block selectors that reference the old pod range. The migration is operationally a new cluster provisioning and workload migration, not an in-place network reconfiguration. I budget 2-4 weeks for a production migration including validation.

## Q: At 18 months, the autoscaler cannot add nodes and logs "no pod CIDR ranges available." Diagnose and fix without cluster recreation.

**Answer:**

This is pod secondary range exhaustion — the nodeIpamController cannot allocate a `/24` block for new nodes because all blocks in the secondary range have been assigned. The diagnosis sequence: first, run `kubectl get nodes -o json | jq -r '.items[] | [.metadata.name, .spec.podCIDR] | @tsv'` to count nodes with CIDRs assigned and verify the math — if the pod secondary range is `10.100.0.0/16` (65,536 IPs) with `/24`-per-node allocation, the max is 256 nodes; if I count 256 rows with assigned CIDRs, the range is full. Second, look for nodes with low pod counts via `kubectl get pods -A -o wide | awk '{print $8}' | sort | uniq -c | sort -rn` — any node with fewer than 10 pods running has 246+ IPs locked up in its `/24` that are unused. Draining and deleting those sparse nodes returns their CIDR blocks to the pool immediately and may resolve the immediate autoscaler blockage. If that is not enough, the medium-term fix without cluster recreation is Multi-CIDR, available in GKE 1.26+. I add a new secondary range to the existing subnet (a non-overlapping block, e.g., `10.104.0.0/14` if `10.100.0.0/14` is the existing one), then update the GKE cluster to enable Multi-CIDR mode referencing both secondary ranges. From that point, the nodeIpamController allocates from either range when provisioning new nodes. For the long term, I update the cluster's CIDR planning documentation and add a monitoring alert: when allocated `/24` blocks exceed 80% of the secondary range capacity, trigger a review. The alert formula is `(num_nodes / (secondary_range_size / 256)) > 0.8`.

## Q: Explain ingress traffic via NEG versus NodePort. Walk through the network layer difference.

**Answer:**

NEG (Network Endpoint Group) and NodePort are two fundamentally different models for routing external traffic into a GKE cluster, and they differ in where the DNAT from external load balancer VIP to pod IP happens. With NodePort: the GCP load balancer sends traffic to a node IP on the NodePort (e.g., `10.0.0.5:32080`). kube-proxy's iptables rules on that node DNAT the traffic to a pod IP and port (`10.100.8.3:8080`), which may or may not be local to that node. If the selected pod is on a different node, the traffic hairpins — it leaves the current node for the destination node, traversing the VPC fabric a second time. The hairpin adds latency and doubles the network hops. With NEG: when you create a Service of type `ClusterIP` with a NEG annotation, GKE registers each pod endpoint directly as a backend in a GCP NEG, and the GCP load balancer routes traffic directly to the pod IP and port. No NodePort is involved. No iptables DNAT on the receiving node is needed — the traffic lands at the pod's own node via the alias IP routing. This eliminates hairpinning, preserves the client source IP all the way to the pod (no SNAT from kube-proxy), and allows the GCP load balancer to make health-check and session-persistence decisions at the individual pod level rather than the node level. The NEG model is strictly superior for internet-facing production workloads: lower latency, better observability (logs show real client IPs), and finer-grained load balancing. The operational requirement for NEG is that the cluster must be VPC-native (pod IPs must be natively routable in the VPC), which is another reason to prefer VPC-native over routes-based clusters.

## Q: What are the most dangerous CIDR planning mistakes and how do you prevent them?

**Answer:**

The three most dangerous mistakes, in order of recovery cost. First, undersizing the pod secondary range. This is recoverable with Multi-CIDR in GKE 1.26+ but caused production outages for years before that feature existed. The typical mistake is choosing a `/16` (256 nodes at `/24`-per-node) for a cluster expected to stay at 100 nodes — then autoscaling, node pool upgrades, and organic growth push it to 240 nodes over 18 months, leaving 16 nodes of headroom, and one bad traffic event triggers autoscaling beyond the limit. The fix is to always plan for 2x steady-state node count plus one node pool worth of headroom for in-flight upgrades. Second, CIDR overlap across VPC-peered clusters. Two teams each design their clusters independently and copy the same "standard" CIDR template. Both clusters get `10.100.0.0/14` for pods. Six months later, a platform team tries to peer the VPCs for a cross-cluster service and the peering fails or creates a routing black hole. The fix is a central CIDR registry — a Terraform variable file or spreadsheet tracked in version control — that any new cluster provisioning must consult and update. Third, forgetting the service secondary range. Teams spend all their time sizing the pod range correctly and use a small `/24` (256 addresses) for services because "we won't have 256 services." A `/24` ClusterIP range fills within a year on a busy cluster with many microservices, Helm releases each creating multiple services, and feature flags creating temporary Services. The fix is to always use `/20` (4,096 addresses) for the service range regardless of the expected service count — it costs nothing and prevents an emergency that requires cluster recreation.

## Q: How would you debug a situation where pods across the cluster intermittently fail to start with "network not ready" errors?

**Answer:**

"Network not ready" during pod startup maps to the CNI plugin failing — the kubelet calls the CNI plugin binary to set up the pod network namespace, and the CNI plugin is returning an error. The diagnostic tree: first, `kubectl describe pod <failing-pod>` to get the exact error message from the `FailedCreatePodSandBox` event — the CNI error is included verbatim in this event. Second, check the CNI plugin's own logs; in GKE, the CNI DaemonSet logs are in `kubectl logs -n kube-system -l k8s-app=gke-node-config` or similar depending on the CNI variant. Common error patterns: `"no IP addresses available in range"` means the node's `/24` pod CIDR is fully allocated — all 256 IPs are in use (remember, IPs are not freed until the pod is fully deleted, not just terminated; stuck `Terminating` pods hold IPs). `"failed to allocate for range 0: no IP addresses available"` from host-local IPAM has the same meaning. `"plugin failed to add"` with an IP assignment error may mean the CNI plugin cannot read the node's `spec.podCIDR` from the API server — check controller-manager connectivity. Third, count IPs in use on the affected nodes: `kubectl get pods -A -o wide | grep <node-name> | grep -v Completed | wc -l`. If this is approaching the node's max-pods-per-node limit (default 110), you have hit the per-node cap enforced by the scheduler. If it is approaching 254, you have filled the `/24`. The fix for per-node exhaustion is to either reschedule some pods to other nodes (edit node affinity rules or drain) or increase max-pods-per-node on a new node pool (which also changes the per-node allocation from `/24` to potentially `/26`). The fix for secondary range exhaustion is the Multi-CIDR path described elsewhere.

## Q: A compliance team asks you to ensure that backend servers can always see the originating pod IP, not a NAT address. What architecture changes does this require?

**Answer:**

By default, traffic leaving the cluster goes through at least one SNAT — either the node-level iptables MASQUERADE rule that rewrites pod IP to node IP, or Cloud NAT if the node also lacks a public IP. To preserve the pod source IP end-to-end to an external backend, both layers of SNAT must be bypassed. The first step is the ip-masq-agent configuration: the ip-masq-agent DaemonSet reads a ConfigMap that lists CIDRs that should be excluded from MASQUERADE — traffic to those destination CIDRs leaves the node with the original pod source IP unchanged. If the backend servers are in an on-premises network reachable via Cloud VPN or Interconnect and have a known CIDR (e.g., `192.168.100.0/24`), add that CIDR to the nonMasqueradeCIDRs list in the ip-masq-agent ConfigMap. The cluster must be VPC-native so the pod IP is a real routable VPC address. The VPN or Interconnect must advertise the pod secondary range (e.g., `10.100.0.0/14`) to the on-premises router so that return traffic from the backend server destined to a pod IP knows to route back through the VPN. Cloud NAT must be excluded for traffic to the on-premises range (Cloud NAT is only applied to traffic with no other routing path, so with proper VPN routing this is automatic). With these three pieces in place — ip-masq-agent exclusion for the backend's CIDR, VPC-native pod IPs, and VPN route advertisement of the pod CIDR — the backend server sees the original pod IP as the TCP source, enabling pod-level logging, firewall rules scoped to pod IPs, and compliance audit trails that map to individual workloads.

## Q: How does GKE handle node pool upgrades and what is the IP implication?

**Answer:**

A GKE node pool upgrade is a rolling replacement: GKE cordons and drains each node, deletes it, provisions a new VM with the upgraded node image, and waits for the new node to join and become `Ready` before moving to the next node. The pod CIDR implication is that during the upgrade window, both old and new nodes briefly coexist. The old node is draining but still holds its `/24` block; the new node is provisioned and gets a new `/24` block allocated by the nodeIpamController. At the peak of the upgrade — before the drained node is deleted — there are temporarily N+1 nodes worth of pod CIDRs consumed, where N is the number of nodes in the pool. On a surge upgrade where multiple nodes are upgraded concurrently (controlled by `maxSurge`), the peak consumption is N+maxSurge. If the pod secondary range does not have headroom for maxSurge additional blocks, the surge nodes cannot get pod CIDRs and the upgrade stalls waiting for CIDR availability. This is why the 2x headroom recommendation accounts not just for autoscaling but also for upgrade headroom. In practice: for a 500-node cluster with a default maxSurge of 1, the upgrade peak is 501 nodes worth of `/24` blocks. For a `/14` pod range (1,024 maximum blocks), this is fine. For a `/16` pod range (256 maximum blocks) and a 250-node cluster, maxSurge=1 during an upgrade pushes to 251 blocks — barely within the 256-block limit, with no room for autoscaling simultaneously occurring. I set alerts to fire when utilized `/24` blocks exceed 80% of the secondary range capacity, which catches this scenario before it becomes an outage.
