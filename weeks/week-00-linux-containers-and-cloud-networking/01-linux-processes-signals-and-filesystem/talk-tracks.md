# Talk Tracks

## Q: Explain Linux processes, signals, and the filesystem in one minute.

**Answer:**

Linux exposes all system resources through three kernel abstractions: the process table, the virtual filesystem, and the permission model. Everything a container does is a sequence of syscalls against these three layers — there is no magic, no abstraction that bypasses the kernel. Processes are created via fork(), which duplicates the calling process using copy-on-write page tables so no physical memory is copied until a write fault occurs, and then exec(), which replaces the duplicated address space with a new binary by loading ELF segments through the page cache. clone() is the generalization of fork() that accepts namespace flags — CLONE_NEWPID, CLONE_NEWNS, CLONE_NEWNET — and is how container runtimes establish isolation boundaries. Signals are asynchronous kernel notifications: the kernel sets a bit in task_struct.pending.signal when a signal arrives, and delivers it the next time the process returns from a syscall to user space. SIGTERM is catchable and ignorable — if PID 1 is a shell script with no handler, it is silently discarded and the 30-second grace period expires before SIGKILL forces termination. The Virtual Filesystem provides a uniform open/read/write interface across ext4, overlayfs, tmpfs, procfs, and every other filesystem type, with a dentry cache for path components and a page cache for file data. Inodes are the kernel's per-file metadata structures; a filesystem can exhaust its fixed inode table while still having free block space, causing ENOSPC failures that df -h misses entirely. Linux capabilities decompose root privilege into named units — CAP_NET_ADMIN for network configuration, CAP_SYS_ADMIN for mounting and namespaces — and container runtimes drop most of them by default, so a root process in a container will still receive EPERM for syscalls that require capabilities not in its effective set.

## Q: docker stop takes 10 seconds every time — what's happening?

**Answer:**

When `docker stop` is called, the Docker daemon sends SIGTERM to PID 1 of the container, waits 10 seconds (the default stop timeout), and then sends SIGKILL. The 10-second delay means PID 1 is not handling SIGTERM. The most common cause is an entrypoint that uses the shell form rather than the exec form in the Dockerfile. If the Dockerfile says `ENTRYPOINT ["sh", "-c", "exec java -jar app.jar"]` or `CMD java -jar app.jar`, the container runtime starts `/bin/sh -c "java -jar app.jar"` as PID 1. The kernel's POSIX shell implementation ignores SIGTERM by default when running non-interactively — this is specified behavior for shells, not a bug. The Java process is a child of the shell at PID 2 or higher, and the shell does not forward SIGTERM to its children. The application never receives the signal. The fix is to ensure the application binary is PID 1 directly: use the JSON exec form `ENTRYPOINT ["java", "-jar", "app.jar"]`, or if a shell wrapper is mandatory, use `exec` to replace the shell process — `exec java -jar app.jar` calls the execve() syscall which replaces the shell with the Java process at the same PID, so the Java process is now PID 1 and receives SIGTERM directly. You can verify the problem by running `kubectl exec <pod> -- cat /proc/1/cmdline | tr '\0' ' '` — if the output starts with `/bin/sh`, you have a shell at PID 1. Check `grep SigCgt /proc/1/status` to see the signal handler bitmask — bit 14 (value 0x4000) being zero means no SIGTERM handler is registered.

## Q: Walk me through fork() to the first instruction of the child process.

**Answer:**

The sequence from fork() to the first child instruction involves five distinct kernel operations. First, the parent calls fork() — the kernel allocates a new task_struct for the child, assigns the next available PID in the current PID namespace, and clones the parent's memory management descriptor. The page table is duplicated: both parent and child now share the same physical pages, but every writable page's page table entry is marked copy-on-write read-only in both mappings. The child is placed on the CPU run queue with task_struct.state set to TASK_RUNNING. fork() returns the child's PID to the parent and returns 0 to the child — both continue executing from the instruction after the fork() call. Second, any write to a shared page in either process triggers a page fault. The fault handler detects the COW flag on the virtual memory area, allocates a new physical page, copies the contents of the original page into the new page, updates the faulting process's page table entry to point to the new page with write permissions, and returns — the write proceeds. The other process still maps the original page. Third, the child calls execve() with the path to the new binary. The kernel validates the path, checks execute permission, reads the ELF header, and then tears down all current VMAs — the entire previous address space is unmapped. Signal handlers are reset to SIG_DFL because the code they pointed to is gone. Fourth, the kernel creates new VMAs for the ELF text, data, and BSS segments, and sets up the initial stack with argc, argv, and envp. The ELF entry point is typically the dynamic linker, ld.so. Fifth, ld.so loads required shared libraries into the address space via mmap(), resolves PLT/GOT relocations for cross-library function calls, runs each library's constructor, and jumps to main(). The first instruction of the application's main() function executes. The total process keeps the same PID throughout but has an entirely replaced address space.

## Q: Node reports disk full but df -h shows 60% used — what else do you check?

**Answer:**

When writes fail with ENOSPC but df -h shows substantial free space, the filesystem has exhausted its inode table, not its block space. Run `df -i` immediately — this reports inode usage per filesystem. Any filesystem at 100% in the `IUse%` column cannot create new files regardless of free blocks. Inodes are the kernel's per-file metadata structures, and on ext4 the inode table size is fixed at format time using the `-i bytes-per-inode` ratio. The default of one inode per 16KB works well for typical workloads but is badly undersized for container workloads that unpack thousands of small files. The most likely culprits on a Kubernetes node are the container overlay filesystem on `/var/lib/containerd` or `/var/lib/docker`, where each container's writable upper layer consumes inodes on the host filesystem for every file written into the container. A Python application that installs a virtualenv creates thousands of `.pyc` files; a Node.js build creates tens of thousands of files in `node_modules`. To find the inode consumers, run `find / -xdev -printf '%h\n' 2>/dev/null | sort | uniq -c | sort -rn | head -20` — this counts the number of entries per directory across the entire filesystem. The fix depends on severity: for immediate relief, clean up the overlay layers using `crictl rmi --prune` or `docker system prune` to remove unused image layers; for structural fix, reformat the volume with a smaller inode ratio (`mkfs.ext4 -i 4096`) or switch to XFS which allocates inodes dynamically. Also configure the kubelet with `--eviction-hard=nodefs.inodesFree<5%` — by default the kubelet only watches block space, not inode space, so nodes can silently hit inode exhaustion without triggering eviction or DiskPressure.

## Q: Explain CAP_NET_ADMIN — when does a workload need it?

**Answer:**

CAP_NET_ADMIN is the Linux capability that grants permission to perform network administration operations: configuring network interfaces (setting IP addresses, flags, MTU via SIOCSIFADDR and SIOCSIFFLAGS ioctls), manipulating the routing table (add/delete routes via RTNETLINK), managing iptables and nftables rules (which require CAP_NET_ADMIN at the netlink socket level), creating and configuring TUN/TAP devices, setting socket options that require elevated privilege such as SO_BINDTODEVICE for certain interface binding operations, and enabling promiscuous mode on network interfaces. Without CAP_NET_ADMIN, these operations fail with EPERM even for UID 0 processes because Linux capabilities and UID are orthogonal — the capability check in the kernel happens independently of the traditional root UID check. In Kubernetes, the main users of CAP_NET_ADMIN are: the Istio istio-init init container (which sets iptables rules to intercept pod traffic before the Envoy sidecar starts), CNI plugins run as DaemonSets that configure pod network interfaces and routes on each node, Calico and Cilium agents that manage BGP and eBPF network policies, and any monitoring agent that uses raw sockets or packet capture. The correct pattern is to grant CAP_NET_ADMIN in the pod's securityContext.capabilities.add field rather than setting privileged: true — privileged re-grants all capabilities and also bypasses seccomp and AppArmor profiles, which is a much larger security surface. When auditing a workload, check `grep CapEff /proc/1/status` inside the container and decode the bitmask with `capsh --decode=<value>` to verify that only the required capabilities are present.

## Q: kubectl exec vs nsenter — what's different at the kernel level?

**Answer:**

Both kubectl exec and nsenter allow you to run a command inside a container's namespaces, but they operate at fundamentally different levels of the stack. kubectl exec sends an Exec request to the Kubernetes API server, which forwards it to the kubelet on the node. The kubelet uses the container runtime (containerd or CRI-O) to call the container runtime interface's ExecSync or Exec method. The runtime uses the low-level OCI runtime (runc) to create a new process inside the container — specifically, runc reads the container's namespace file descriptors from /proc/<container-pid>/ns/ and calls setns() for each namespace (pid, mnt, net, uts, ipc) before exec-ing the command. The new process joins all six namespaces of the container. It is subject to the container's seccomp profile, AppArmor policy, and cgroup membership. kubectl exec is the right tool for interactive debugging from a Kubernetes context. nsenter, run directly on the node, is lower-level: you give it a target PID and flags for which namespaces to enter (-m for mount, -n for net, -p for pid, -u for uts). It calls setns() directly for each requested namespace, then exec-s the command. The key differences: nsenter does not go through the container runtime and is not subject to the container's seccomp profile or AppArmor policy (you are running with the host's security context); nsenter can enter only a subset of namespaces (e.g., just the network namespace, while keeping the host mount namespace), which is useful for debugging network configuration without being constrained by the container filesystem; and nsenter requires host-level access (typically via `kubectl debug node/` or SSH to the node). Use kubectl exec for routine debugging; use nsenter when kubectl exec fails because the container lacks debugging tools, the container is in CrashLoopBackOff, or you need host-level visibility into a container's network state.

## Q: What can go wrong?

**Answer:**

Four failure modes dominate in production. First, PID 1 signal handling: a shell wrapper as PID 1 ignores SIGTERM, causing every pod termination to take the full grace period and every rolling update to be maximally slow — identify it with `cat /proc/1/cmdline` and fix it with the exec ENTRYPOINT form or the `exec` shell builtin. Second, zombie accumulation: any container whose entrypoint spawns subprocesses without waiting for them creates zombies that consume PID namespace slots — identify with `ps aux | grep ' Z '` and fix with tini or dumb-init as PID 1. Third, inode exhaustion: a filesystem with a small inode table and many small files reports ENOSPC while `df -h` shows ample space — identify with `df -i` and fix by reformatting with a smaller inode ratio or using XFS, and add `--eviction-hard=nodefs.inodesFree<5%` to the kubelet. Fourth, capability mismatch: a UID 0 process fails with EPERM because the container runtime dropped the required capability from the bounding set — identify with `grep CapEff /proc/1/status` and `capsh --decode`, and fix by explicitly adding the needed capability in the pod's securityContext rather than reaching for privileged: true. A fifth non-obvious failure is the D-state (uninterruptible sleep) process: a container with a process blocked on a hung NFS or iSCSI mount cannot be killed, cannot receive signals, and will block `kubectl delete pod` indefinitely — the only resolution without a node reboot is to unmount the hanging filesystem from the host.

## Q: How would you debug it?

**Answer:**

My debugging approach starts with the /proc virtual filesystem because it exposes the kernel's complete view of every process without requiring any additional tooling. For any process, `/proc/<pid>/status` gives the state (R/S/D/Z/T), the signal masks (SigBlk, SigCgt — pending, blocked, and caught signals as hex bitmasks), and the capability sets (CapPrm, CapEff, CapBnd). `/proc/<pid>/cmdline` gives the full command line with NUL-separated arguments. `/proc/<pid>/fd/` lists all open file descriptors as symlinks to their targets — useful for detecting leaked file handles or verifying which sockets are open. `/proc/<pid>/maps` shows the complete virtual address space: code, heap, stack, mmap'd files, and shared libraries. `/proc/<pid>/cgroup` shows cgroup membership, which maps back to which container the process belongs to. For signal debugging, I check `SigCgt` in `/proc/<pid>/status` — a bitmask where each bit corresponds to a signal number; if bit N is set, the process has a handler for signal N+1. SIGTERM is signal 15, so bit 14 (value 0x4000) in SigCgt indicates a SIGTERM handler. For capability debugging, I decode `CapEff` with `capsh --decode=<hex>` to get the human-readable list of effective capabilities, then cross-reference with the syscall that is failing. For filesystem issues, `df -i` is the first command, followed by `stat -f <path>` for detailed inode statistics and `find / -xdev -printf '%h\n' | sort | uniq -c | sort -rn | head -20` to identify inode consumers. For signal forwarding verification on a live container, `strace -e trace=signal -f -p <pid>` shows every signal operation in real time.

## Q: How do Linux namespaces connect to container isolation, and where do they break down?

**Answer:**

Linux namespaces partition the kernel's global resource namespaces so that a set of processes sees an isolated view of specific resources. The six namespaces used by containers are: PID (process ID numbering — PID 1 inside the container is a different kernel task_struct from host PID 1, and the container cannot see or signal processes outside its PID namespace), mount (each container gets its own mount table, so bind mounts and overlay mounts are per-container without affecting the host hierarchy), network (each container gets its own network stack — loopback device, routing table, iptables rules, sockets — connected to the host via a veth pair), UTS (hostname and NIS domain name), IPC (System V message queues, semaphores, and POSIX shared memory segments), and user (UID/GID mapping, allowing a process that is root in the container to be an unprivileged UID on the host). Namespaces provide isolation but not security in the strong sense — they are in the same kernel and share the same kernel code paths. Namespace isolation breaks down in several important ways: the host filesystem is accessible through privileged volume mounts; the kernel syscall interface is shared, so a kernel vulnerability exploitable from inside a container compromises the host; procfs in the container exposes information about the container's own processes but the host /proc is accessible if the container mounts it; and cgroups limit resource usage but do not prevent a process from trying to exhaust resources (the kernel still processes the attempt, just limits the effect). This is why seccomp (filtering the syscall interface), AppArmor/SELinux (MAC on operations), and capabilities (limiting what root can do) are defense-in-depth layers on top of namespace isolation, not replacements for it. Containers are isolated, not sandboxed — the security boundary assumption is different from that of a virtual machine, which has a separate kernel.

## Q: Explain the /proc filesystem — what can you learn from it without any external tools?

**Answer:**

The /proc virtual filesystem is the kernel's self-reporting interface. It is a pseudo-filesystem — files in /proc do not exist on disk; reading them causes the kernel to generate content dynamically from in-kernel data structures. For any running process at PID N, `/proc/N/` contains: `status` — a human-readable summary of the process state, UID/GID, memory usage (VmRSS for resident set size, VmSwap, VmPeak), thread count, and all six signal bitmasks (SigPnd, SigBlk, SigCgt, SigIgn) and capability bitmasks (CapPrm, CapEff, CapBnd, CapAmb); `cmdline` — the full command line with NUL-byte delimiters between arguments (readable with `tr '\0' ' '`); `environ` — the environment variable list in the same NUL-delimited format; `fd/` — a directory of symlinks from file descriptor numbers to the actual resource (a socket shows as `socket:[inode]`, a file shows as its path, a pipe shows as `pipe:[inode]`); `maps` — the virtual memory map with start/end addresses, permissions (r/w/x/p or r/w/x/s for shared), offset, device, inode, and pathname for every mapping; `net/tcp`, `net/udp`, `net/tcp6` — all active socket connections in hex format, including local and remote address, state (0A = LISTEN, 01 = ESTABLISHED), and inode number; `cgroup` — which cgroup hierarchy and cgroup path the process belongs to; `ns/` — symlinks to the namespace inodes (comparing the inode numbers of `/proc/N/ns/pid` across two processes tells you if they share a PID namespace); and `oom_score` and `oom_adj` — the OOM killer score and adjustment for memory pressure eviction. Running `/proc/meminfo` gives system-wide memory statistics. `/proc/sys/` contains every kernel tunable parameter organized by subsystem, and writing to these files applies the parameter change at runtime — this is how `sysctl -w` works under the hood. For container debugging without exec access to a pod, accessing `/proc/<container-pid>/` from the host gives full visibility into the container's process state, open files, and network connections.
