# Talk Tracks

## Q: Explain container networking in one minute.

**Answer:**

Container networking in Kubernetes rests on three kernel primitives and one architectural contract. The kernel primitives are network namespaces — which give each pod an isolated network stack with its own IP, routing table, and iptables rules — veth pairs, which are virtual Ethernet cables that connect a pod's isolated namespace to the host's root namespace, and Linux bridges, which are software Layer 2 switches that forward frames between all veth endpoints on the same node. The architectural contract is the Kubernetes pod networking model: every pod gets a unique routable IP, pods can reach each other without NAT, and agents on a node can reach all pods on that node. CNI — Container Network Interface — is the mechanism that binds these together. When kubelet creates a pod, it hands the pod's network namespace file path to the CNI binary on stdin. The CNI binary creates the veth pair, plugs one end into the pod netns and one end into the bridge, calls the IPAM plugin to allocate an IP, assigns that IP to eth0 inside the pod, and writes the default route. From that point, the kernel's routing and bridging code handles packet forwarding. Cross-node traffic is handled by either encapsulation (Flannel VXLAN wraps pod packets in UDP before sending to the remote node's IP) or by routing (Calico BGP advertises pod CIDRs so the physical network routes pod packets natively). kube-proxy sits alongside CNI and programs iptables DNAT rules to translate Service ClusterIPs to real pod IPs, with conntrack tracking the translation for symmetric reply routing.

## Q: Trace a TCP packet from pod A on node 1 to pod B on node 2 through VXLAN — name every kernel structure it traverses.

**Answer:**

Starting inside pod-a's network namespace: the application calls `connect()` to pod-b's IP `10.244.1.10`. The kernel builds an IP packet with src `10.244.0.10` and dst `10.244.1.10`. The pod netns routing table has one route: `default via 10.244.0.1 dev eth0`. Since `10.244.1.10` is outside the local /24, the packet is sent to the default gateway `10.244.0.1`. The kernel does an ARP lookup for `10.244.0.1` — the bridge `cni0` responds (ARP proxy). The packet enters `eth0` in the pod netns, traverses the veth pair kernel ring buffer, and emerges on the host-side veth (`vethXXX`). Because `vethXXX` is a slave port of the `cni0` bridge, the bridge receives the Ethernet frame. The bridge looks up the destination IP `10.244.1.10` in its routing context — it's not local — so it passes the frame up to the host kernel's IP routing stack. The host routing table, populated by flanneld, has `10.244.1.0/24 dev flannel.1 onlink`. The packet is handed to the `flannel.1` VXLAN interface. `flannel.1` is a VXLAN VTEP: it looks up the FDB (Forwarding Database) for the MAC entry corresponding to `10.244.1.0` VTEP on node-2. The FDB entry maps that MAC to node-2's underlay IP `192.168.1.2`. The VXLAN driver builds the encapsulation: original IP frame gets wrapped in an Ethernet frame (inner), then a VXLAN header (VNI=1), then UDP (dst port 8472), then an outer IP header (src `192.168.1.1`, dst `192.168.1.2`). The outer IP packet goes through the host iptables POSTROUTING chain (no NAT applies for pod-to-pod), out the physical NIC `eth0`, and across the VPC to node-2. Node-2's kernel receives the UDP packet on port 8472 at its physical NIC. The `flannel.1` VXLAN driver on node-2 recognizes the VNI, strips all VXLAN/UDP/IP headers, and delivers the original IP packet (src `10.244.0.10`, dst `10.244.1.10`) to node-2's routing stack. Node-2's routing table has `10.244.1.0/24 dev cni0`, so the packet is forwarded to the bridge, then to pod-b's veth, then to pod-b's `eth0`. The application in pod-b receives the TCP SYN. From pod-b's perspective, the packet arrived directly from pod-a with no NAT — the VXLAN encapsulation was invisible to both endpoints.

## Q: What is the Kubernetes pod networking model and why does it matter for applications?

**Answer:**

The Kubernetes pod networking model specifies three invariants: every pod gets a unique IP address that is routable within the cluster; pods on any node can communicate directly with pods on any other node using those IPs without any form of NAT; and nodes can reach any pod directly. These invariants were a deliberate design choice to solve the operational complexity that plagued earlier container platforms like Mesos Marathon and Docker Swarm, where containers shared node IPs and were differentiated only by port mappings. Port mapping creates two hard problems: port conflict across tenants (two services can't both listen on port 8080 on the same node) and NAT traversal complexity (the address a container binds inside its namespace is different from the address clients use to reach it, breaking applications that embed their own IP in protocol payloads — like Cassandra gossip, etcd peer election, and gRPC server reflection). The Kubernetes model eliminates both problems. Applications can bind on any port without conflict, because each pod has its own IP. Applications can assume the IP they bind is the IP clients connect to — there is no NAT rewrite in the forwarding path between pods. This also makes identity-based security policies tractable: a NetworkPolicy that says "allow traffic from pod with label app=frontend to port 5432" is implementable by IP because pod IPs are stable per-pod identity for the lifetime of the pod. Where the model gets subtle is that it says nothing about how this is implemented — Flannel VXLAN, Calico BGP, AWS VPC CNI, and Cilium all satisfy the invariants via different mechanisms — which is why CNI exists as a pluggable interface rather than a built-in kernel feature.

## Q: Trace what happens when a pod connects to a Service ClusterIP — every NAT step and conntrack entry.

**Answer:**

When pod-a (`10.244.0.10`) connects to Service `10.96.0.100:80`, which has two endpoints `10.244.0.20:8080` and `10.244.1.20:8080`, the TCP SYN packet (src `10.244.0.10`, dst `10.96.0.100:80`) leaves pod-a's `eth0`, traverses the veth pair, and enters the host root network namespace. Before the packet is routed, Netfilter fires the `PREROUTING` hook on the `nat` table. kube-proxy has installed a chain `KUBE-SERVICES` in `PREROUTING`. The packet matches the rule for `10.96.0.100:80` and jumps to `KUBE-SVC-XXXXXX`. That chain uses iptables `statistic --mode random --probability 0.5` to randomly select one endpoint. Suppose it selects `10.244.1.20:8080`. An iptables DNAT rule rewrites the destination: the packet's dst IP changes from `10.96.0.100` to `10.244.1.20`, dst port from `80` to `8080`. Conntrack records this translation as a new connection: original tuple `(10.244.0.10:EPHEMERAL, 10.96.0.100:80)` maps to translated tuple `(10.244.0.10:EPHEMERAL, 10.244.1.20:8080)`. Now the IP routing stack sees a packet destined for `10.244.1.20`, which is on node-2. The packet follows the cross-node VXLAN path. pod-b on node-2 receives a connection from `10.244.0.10:EPHEMERAL` to `10.244.1.20:8080` — it has no idea a Service VIP was involved. pod-b's SYN-ACK goes back to `10.244.0.10`. On node-1, the reply packet (src `10.244.1.20:8080`, dst `10.244.0.10:EPHEMERAL`) enters the host netns. Netfilter's conntrack at `PREROUTING` finds the connection entry and automatically reverse-translates: dst `10.244.0.10` unchanged, src rewrites from `10.244.1.20:8080` to `10.96.0.100:80`. The application in pod-a sees the reply arriving from `10.96.0.100:80` — the Service VIP — maintaining the abstraction. The MASQUERADE case is when the selected endpoint is on the same node: kube-proxy marks the packet with `KUBE-MARK-MASQ`, and in the `POSTROUTING` chain, that mark triggers SNAT of the source from `10.244.0.10` to the bridge IP `10.244.0.1`. This ensures the reply from the local pod goes through the bridge and conntrack, rather than being delivered directly from pod-to-pod bypassing the de-NAT path.

## Q: Why does Istio use iptables REDIRECT inside the pod network namespace? What is the alternative?

**Answer:**

Istio's sidecar proxy (Envoy) must intercept all inbound and outbound traffic for the application container without requiring the application to know about the proxy. The mechanism is iptables REDIRECT rules installed inside the pod's network namespace by the `istio-init` init container (or by the CNI plugin in ambient mode). The `istio-init` container runs with `NET_ADMIN` capability, which allows it to modify the pod's own iptables (remember: iptables rules are per-network-namespace, so modifications inside the pod netns do not affect the node or other pods). It writes two rules: in the `PREROUTING` chain of the `nat` table, all inbound TCP traffic (except on port 15090, 15021 for health checks) is redirected to Envoy's inbound listener on port 15006 using REDIRECT — REDIRECT is a special NAT target that rewrites the destination to the local socket listening on the given port, and crucially sets `SO_ORIGINAL_DST` on the socket so the proxying process can call `getsockopt(SO_ORIGINAL_DST)` to discover the original destination the application intended. In the `OUTPUT` chain, all outbound TCP traffic from the pod (except from Envoy's own UID 1337) is redirected to Envoy's outbound listener on port 15001. Envoy reads the original destination via `SO_ORIGINAL_DST`, applies L7 policy and observability, and then opens a real connection to the original destination using its own UID (1337), which is exempted from the OUTPUT redirect rule to avoid an infinite loop. The alternative is eBPF-based traffic interception, which Istio ambient mode uses: Cilium or Istio's own eBPF programs intercept packets at TC hooks on the veth interface before they enter the pod netns, eliminating the need for iptables inside the pod and removing the `istio-init` container requirement. The eBPF approach requires fewer privileges (no `NET_ADMIN` per pod), has lower overhead, but requires a more recent kernel (5.10+) and a CNI that cooperates with the eBPF programs.

## Q: A new Cilium node cannot reach pods on old nodes in the cluster — what do you check?

**Answer:**

The most likely cause is that the new node's Cilium agent has not established its identity and policy in the Cilium control plane, causing the eBPF identity maps on both old and new nodes to have no entry for the new node's pods. Cilium uses its own identity model layered on top of IP: each pod is assigned a numeric identity based on its labels, and eBPF programs enforce policy by looking up packet source/destination IPs in an identity map. If the Cilium operator or controller has not propagated the new node's CiliumNode resource and its pod CIDRs, old nodes' eBPF maps have no route entry for the new CIDR and silently drop packets. Start by checking `kubectl get ciliumnodes` — the new node must appear with its `ipv4-pod-cidr` correctly populated. Then check `kubectl exec -n kube-system ds/cilium -- cilium status` on a pod on an old node: look for `KVStore` connectivity status (if using etcd backend) and check `ClusterMesh` if multicluster is involved. Run `cilium bpf ipcache list` on an old node and verify the new pod's IP is present with a valid identity. If it is missing, the issue is in the Cilium operator not propagating the CiliumNode object, which means the operator pod itself may be crashing or restarting — check `kubectl logs -n kube-system deploy/cilium-operator`. Also verify that VXLAN (if using tunnel mode) is configured on the new node: `ip link show cilium_vxlan` should be present. For BGP mode, check `kubectl exec -n kube-system ds/cilium -- cilium bgp peers` to confirm the new node has peered with its neighbor. Finally, use `cilium monitor --type drop` on a source pod's node to see if packets to the new node's pods are being dropped at the eBPF level and which drop reason is recorded (e.g., `POLICY_DENIED` vs `NO_TUNNEL_ENDPOINT`).

## Q: What can go wrong with container networking, and what are the highest-impact failure modes?

**Answer:**

The highest-blast-radius failure mode is CNI DaemonSet disruption during a node upgrade wave. If the CNI DaemonSet — Calico node, Flannel — is not running on a node, the CNI binary is absent and every pod scheduling attempt to that node fails with `ContainerCreating` indefinitely. During rolling node upgrades where 20 nodes are drained simultaneously, if the CNI DaemonSet has a `maxUnavailable: 0` PodDisruptionBudget and the rolling upgrade strategy doesn't tolerate unavailability, the CNI pod may not start on new nodes before kubelet starts scheduling work there, causing a scheduling blackout. The second high-impact failure is podCIDR exhaustion, which is subtle because it's gradual and node-specific. A node running 250 pods against a `/24` CIDR has 4 IPs left. High-churn workloads (CronJobs, batch processing) create pods faster than terminated pod IPs are released, eventually exhausting the pool. The node becomes unable to schedule any new pods while the rest of the cluster is unaffected. The third failure mode is iptables rules corruption or truncation. kube-proxy regenerates rules from Endpoints on every Endpoints change event. In clusters with 10,000 services and high endpoint churn, kube-proxy may spend significant CPU on iptables-restore. During the iptables-restore atomic swap, if it fails midway due to locking or OOM kill of kube-proxy, the nat table may have inconsistent or incomplete rules, causing a percentage of service connections to fail. This manifests as intermittent 5xx errors that self-resolve when kube-proxy completes the next full sync cycle. The fourth failure mode is conntrack table exhaustion: a node handling high connection rates — an ingress gateway running thousands of connections per second — can fill the conntrack table (`nf_conntrack_max`), causing the kernel to log `nf_conntrack: table full, dropping packet` and reject new connections entirely at the NIC level.

## Q: How would you debug intermittent pod-to-pod connectivity failures?

**Answer:**

I approach this systematically by narrowing the failure to a specific layer: pod netns, host forwarding, or cross-node path. First, I confirm the failure scope — is it all pod-to-pod traffic or specific source/destination pairs? `kubectl exec pod-a -- ping 10.244.1.10` and `kubectl exec pod-a -- curl -v http://10.244.1.10:8080` isolate whether it is ICMP, TCP connection, or higher-level HTTP. If ping works but TCP fails, conntrack or iptables filtering is more likely than a routing issue. Second, I check the CNI health on both involved nodes: `kubectl get pods -n kube-system -l k8s-app=calico-node -o wide` to confirm the CNI DaemonSet is running, and inspect its logs for errors. On the node itself, `ip route show` confirms pod routes are present; `ip link show cni0` or `ip link show flannel.1` confirms the bridge and tunnel interfaces are up. Third, I use `tcpdump` to trace the packet path: on the source node I capture on the source pod's veth (`tcpdump -i vethXXX -nn tcp port 8080`) to verify the packet leaves the pod. Then on the `cni0` bridge to verify it enters the host stack. Then on `flannel.1` or `calico.1` to verify encapsulation. On the destination node I capture in reverse order: physical NIC, then tunnel interface, then bridge, then destination pod's veth. Any interface where the packet arrives but doesn't leave points to the failing layer. For iptables policy-based drops, `iptables -L FORWARD -n -v` shows packet/byte counters per rule — a rule with high matches and `DROP` target is the culprit. For Cilium, `cilium monitor --type drop` streams kernel-level drop events with reasons. For NetworkPolicy conflicts, `kubectl get networkpolicy -A` and `kubectl describe networkpolicy <name>` clarify which policies apply; Calico's `calicoctl get networkpolicy` shows the Calico-level policy evaluation.

## Q: How does eBPF improve on iptables for service load balancing at scale?

**Answer:**

iptables service load balancing in kube-proxy has a fundamental algorithmic problem: rule evaluation is sequential. For a cluster with 5,000 services averaging 3 endpoints each, the KUBE-SERVICES chain has ~5,000 entries, and each endpoint selection within a service adds more rules via `KUBE-SVC-*` chains. A packet hitting a Service ClusterIP must traverse all entries before its rule before a verdict is reached — in the worst case, O(N) rule evaluations per packet. With 100,000 connections per second on a busy ingress node, this saturates one CPU core on iptables rule processing alone, and updating rules (kube-proxy calls `iptables-restore` on every Endpoints change) takes 1-10 seconds during which a full table lock is held. Cilium's eBPF approach replaces the sequential chain with a BPF map lookup. Service VIPs are stored in a BPF hash map (`cilium_lb4_services_v2`) keyed by `(IP, port, protocol)`. A packet entering the TC ingress hook on the veth triggers a BPF program that does a single O(1) hash map lookup to find the service's backend selection algorithm and backend list, then a second O(1) map lookup to select the backend (round-robin state is maintained in a separate per-CPU array map). The DNAT rewrite is applied inline in the BPF program, and the connection is tracked in a per-CPU BPF conntrack map, not the kernel's `nf_conntrack` table. Updates to service backends are reflected in BPF maps atomically using `bpf_map_update_elem()` with `BPF_EXIST` flag — there is no table lock, no `iptables-restore`, and the update applies to new connections immediately without disrupting existing ones. At 10,000 services, Cilium's per-packet overhead for service translation is constant; kube-proxy's grows linearly. Cilium also supports DSR (Direct Server Return) for NodePort services where the backend pod replies directly to the client without hairpinning through the node that received the external packet, reducing server-side bandwidth by ~50% for large responses.

## Q: Walk through what a CNI plugin must do — what inputs does it receive and what must it produce?

**Answer:**

The CNI spec defines a stateless binary protocol. The container runtime (containerd, cri-o) executes the CNI binary after creating the pod's network namespace but before starting the main container process. The runtime provides three mandatory environment variables: `CNI_COMMAND` (always ADD for pod creation), `CNI_NETNS` (the path to the network namespace file, e.g., `/var/run/netns/cni-12345678`), and `CNI_CONTAINERID` (a unique identifier for the container, typically the pause container's ID). The network configuration is provided on stdin as JSON, reading from the file at `/etc/cni/net.d/10-flannel.conflist` or equivalent. For ADD, the plugin must accomplish five things and return a result structure on stdout. First, it creates a veth pair: `ip link add eth0 type veth peer name vethXXXXXX`. Second, it moves one end into the pod's network namespace: `ip link set eth0 netns <CNI_NETNS>` — after this call, `eth0` is visible only inside the pod netns. Third, it calls the IPAM plugin (either a separate binary like `host-local` or `calico-ipam`, or internally) to allocate an IP from the node's podCIDR. The IPAM plugin returns an IP with prefix length (e.g., `10.244.0.15/24`) and the gateway IP (`10.244.0.1`). Fourth, inside the pod netns, the plugin assigns that IP to `eth0`, brings up the interface, and adds the default route via the gateway: `ip netns exec <ns> ip addr add 10.244.0.15/24 dev eth0` and `ip netns exec <ns> ip route add default via 10.244.0.1 dev eth0`. Fifth, back in the host root ns, the plugin attaches the host-side veth to the bridge (`ip link set vethXXXXXX master cni0`) and brings it up. The plugin returns a JSON result on stdout containing the assigned IP, interface name, and gateway. If any step fails, it returns a non-zero exit code and a JSON error on stderr. For DEL (pod deletion), the plugin reverses the process: removes the veth from the bridge, deletes the veth pair (which also removes the pod's `eth0`), and releases the IP back to the IPAM plugin's pool. The entire ADD path executes synchronously before the container starts — the container process has a fully wired network namespace from its first instruction.
